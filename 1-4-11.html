WCAG 1.4.11: Non-text Contrast

View main list

Brief summary: Visual information needed to identify user interface components, component states, and meaningful graphical objects should have at least 3:1 contrast against adjacent colors, with specific exceptions.

View the W3C Understanding page for WCAG 1.4.11 Non-text Contrast.

These examples focus on machine-detectable contrast cases: input boundaries, button boundaries, focus indicators, meaningful icons, and status indicators. Decorative graphics are included as a control example because low-contrast decorative graphics should not automatically fail this success criterion.

Passing examples

Passing example 1: Text input has sufficient boundary contrast

This succeeds because the input's visible border has sufficient contrast against the white page background, making the component boundary identifiable.

Passing example 2: Button has sufficient boundary contrast

This succeeds because the button border has sufficient contrast against the adjacent white background.

Passing example 3: Focus indicator has sufficient contrast

This succeeds because the keyboard focus indicator uses a high-contrast outline that is visible against the white background.

Passing example 4: Meaningful icon has sufficient contrast

This succeeds because the icon communicates meaning and has sufficient contrast against the white background.

Passing example 5: Decorative low-contrast icon is not needed to understand the control

This succeeds because the low-contrast icon is decorative. The visible button text communicates the control's purpose, so the icon itself is not required to identify the component or understand the action.

Failing examples

Failing example 1: Text input boundary has insufficient contrast

This fails because the input's visible border is too light against the white background, making the component boundary difficult to identify.

SiteLint observation: SiteLint does not report this intentionally failing example. The text input boundary is defined with a very light border against a white background, creating a machine-detectable non-text contrast failure candidate.

Recommendation: SiteLint should evaluate input boundaries used to identify form controls and report cases where the boundary has less than 3:1 contrast against the adjacent background.

Failing example 2: Button boundary has insufficient contrast

This fails because the button depends on a visible border to identify the component boundary, but that border has insufficient contrast against the adjacent white background.

SiteLint observation: SiteLint does not report this intentionally failing example. The button relies on a visible border to identify its boundary, but that border has insufficient contrast against the adjacent background.

Recommendation: SiteLint should check visible borders or other boundary indicators on buttons and controls when those indicators are needed to identify the component.

Failing example 3: Focus indicator has insufficient contrast

This fails because the keyboard focus indicator is too light against the white background. To test this manually, tab to the button and inspect the focus outline.

SiteLint observation: SiteLint does not report this intentionally failing example. The focus indicator is defined in CSS with a low-contrast outline. This may require the auditor to evaluate focus-state styles rather than only the default rendered state.

Recommendation: SiteLint should test or simulate focus states for interactive elements and evaluate whether the visible focus indicator meets the required non-text contrast threshold.

Failing example 4: Meaningful icon has insufficient contrast

This fails because the search icon is meaningful and its shape has insufficient contrast against the white background.

SiteLint observation: SiteLint does not report this intentionally failing example. The icon is part of a labeled search control and is visually meaningful, but its SVG fill color has insufficient contrast against the adjacent background.

Recommendation: SiteLint should consider evaluating SVG graphical objects used in controls when the icon contributes to identifying the control or its action. Decorative SVGs should remain excluded where the visible text already provides the necessary information.

Failing example 5: Status indicator has insufficient contrast

This fails because the status dot is a meaningful graphical object and has insufficient contrast against the white background.

SiteLint observation: SiteLint does not report this intentionally failing example. The status indicator is a meaningful graphical object, but the dot has insufficient contrast against the adjacent white background.

Recommendation: SiteLint should consider checking meaningful graphical status indicators for non-text contrast, especially when a shape, icon, or marker is used as part of the visual status presentation.

System status: Offline

Current 1.4.11 coverage takeaway

Recommendation: SiteLint does not currently report any of the intentionally failing non-text contrast examples on this page. This suggests a significant WCAG 1.4.11 coverage gap for common machine-detectable patterns: low-contrast control boundaries, low-contrast focus indicators, low-contrast meaningful SVG icons, and low-contrast status indicators.

The strongest starting point would be control boundaries and focus indicators because those are usually available through computed CSS. Graphical objects may require more context to distinguish meaningful icons from decorative graphics, but common SVG-in-control and status-indicator patterns are still worth targeted evaluation.
