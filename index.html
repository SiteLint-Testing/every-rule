This index tracks the WCAG success-criterion test pages being created to evaluate SiteLint coverage. Each linked page contains representative passing and failing scenarios for that success criterion, followed by observations and recommendations when SiteLint behavior has been reviewed.

Created (41) and skipped (46) WCAG test pages
1 A 1.1.1 Non-text Content
  • No SiteLint issues requiring follow-up identified from the current first-pass test page.
2 A 1.2.1 Audio-only and Video-only (Prerecorded)
  • Caution: This success criterion is only partly testable through static markup. Whether the provided alternative is actually equivalent to the prerecorded audio-only or video-only content requires human review of the media and its alternative.
  • SiteLint incorrectly flags a passing video-only example that provides an associated text alternative, even though WCAG 1.2.1 allows prerecorded video-only content to pass through an alternative for time-based media.
  • The current media rule appears to treat the absence of a track element or audio source as sufficient evidence of failure, which is too narrow for 1.2.1.

Recommendation: 1.2.1 logic should account for the valid text-alternative pathway rather than treating the absence of embedded track/audio markup as conclusive.
3 A 1.2.2 Captions (Prerecorded)
  • Caution: A static page can test whether a captions mechanism appears to be present, but it cannot verify whether the captions are accurate, complete, synchronized, or include relevant non-speech audio.
  • SiteLint incorrectly flags a passing example that is clearly labeled as a media alternative for text, which is an exception under WCAG 1.2.2.
  • SiteLint catches the missing-captions failing scenario only through a finding mapped to 1.2.1, not 1.2.2.
  • SiteLint misses the subtitles-only failing scenario, suggesting that subtitles may be treated as equivalent to captions.

Recommendation: SiteLint needs explicit, accurately mapped 1.2.2 coverage and should not treat kind="subtitles" as a guaranteed substitute for captions.
4 A 1.2.3 Audio Description or Media Alternative (Prerecorded)
  • Caution: This success criterion is only partly testable through static markup. Whether audio description or a media alternative fully conveys the necessary visual and auditory information requires human review of the media content.
  • SiteLint incorrectly flags a passing example that provides a full media alternative in text form.
  • SiteLint incorrectly flags a passing example that is clearly labeled as a media alternative for text.
  • SiteLint flags the genuine failing scenario, but reports it under WCAG 1.2.1 rather than 1.2.3.

Recommendation: SiteLint needs explicit 1.2.3 coverage that accounts for audio description, full media alternatives, and the media-alternative-for-text exception.
5 AA 1.2.4 Captions (Live)
  • Skipped: This success criterion applies to live synchronized media. A meaningful test requires real-time or simulated-live audio/video and a judgment about whether captions are provided as the live content occurs. Reviewing the static HTML only cannot determine whether live captions are present, accurate, synchronized, or complete.
6 AA 1.2.5 Audio Description (Prerecorded)
  • Caution: This success criterion is only partly testable through automation because the need for audio description depends on whether important visual information is already conveyed in the existing audio track.
  • SiteLint incorrectly flags a passing example where important visual information is already conveyed in the main audio track, a valid 1.2.5 scenario.
  • SiteLint flags the genuine failing scenario, but again reports it under WCAG 1.2.1 rather than 1.2.5.

Recommendation: SiteLint should not treat the absence of a descriptions track as a confirmed 1.2.5 failure. This often requires focused manual verification.
7 AAA 1.2.6 Sign Language (Prerecorded)
  • Skipped: This requires reviewing the prerecorded video content to determine whether a sign language interpretation is actually provided and whether it covers the spoken audio. HTML can show that a video exists, but it cannot confirm that sign language interpretation appears in the media.
8 AAA 1.2.7 Extended Audio Description (Prerecorded)
  • Skipped: This requires reviewing the timing and content of the prerecorded media to determine whether ordinary audio description cannot fit into the available pauses and extended audio description is therefore required. That judgment depends on the media timeline, not just the surrounding markup.
9 AAA 1.2.8 Media Alternative (Prerecorded)
  • Caution: This success criterion is only partly testable through static markup. A page can show whether a clearly associated media alternative is present or absent, but whether that alternative fully captures the important visual and auditory information requires human review of the media and the alternative.
  • SiteLint incorrectly flags both passing examples that provide full media alternatives: one for prerecorded synchronized media and one for prerecorded video-only media.
  • SiteLint flags the two genuine failing scenarios, but reports them under WCAG 1.2.1 rather than WCAG 1.2.8.

Recommendation: SiteLint needs explicit, accurately mapped 1.2.8 coverage that accounts for full media alternatives rather than reusing the generic “no track or audio source” media heuristic.
10 AAA 1.2.9 Audio-only (Live)
  • Skipped: This success criterion applies to live audio-only content. A meaningful test requires live or simulated-live audio and a judgment about whether an equivalent real-time text alternative is available. Static HTML cannot confirm the live audio content or the accuracy and timing of its alternative.
11 A 1.3.1 Info and Relationships
  • SiteLint correctly flags the ungrouped radio-button example under WCAG 1.3.1, but the rule message says “Checkboxes should be grouped together” even though the controls are radio buttons.
  • SiteLint correctly flags the data table with visual headers but no programmatic header cells under WCAG 1.3.1.
  • SiteLint flags the visible-but-unassociated form label, but maps the finding to WCAG 3.3.2 rather than 1.3.1.
  • SiteLint also reports a WCAG 2.5.8 target-size issue on the radio buttons, which is unrelated to this 1.3.1 test page and may need to be avoided through test-page styling.

Recommendation: Update the grouped-control rule message to include radio buttons, consider whether form-label relationship failures should also map to 1.3.1, and keep the page focused on machine-detectable structural relationship failures.
12 A 1.3.2 Meaningful Sequence
  • Caution: This success criterion is only partly testable through automation. Automated tests can detect some risky patterns where CSS changes visual order, but whether the changed sequence affects meaning may still require human review.
  • SiteLint does not currently report either failing example on the page, even though both use explicit CSS order values that make the visual order differ from the DOM order.

Recommendation: SiteLint should consider a targeted WCAG 1.3.2 manual-verification rule for flex/grid reordering patterns where source order and visual order may diverge.
13 A 1.3.3 Sensory Characteristics
  • Skipped: This requires deciding whether instructions rely only on sensory characteristics such as shape, color, size, visual location, orientation, or sound. A tool can search for risky words like “red,” “left,” or “round,” but a human still has to read the surrounding content and inspect the referenced control to determine whether a non-sensory identifier is also provided.
14 AA 1.3.4 Orientation
  • Skipped: This requires checking whether the page can be viewed and operated in both portrait and landscape orientation. A meaningful test would need to render or simulate the page in both orientations and determine whether content or functionality is blocked in one of them.
15 AA 1.3.5 Identify Input Purpose
  • SiteLint correctly reports an invalid autocomplete token under WCAG 1.3.5.
  • SiteLint does not report missing autocomplete tokens on common user-information fields, such as full name.
  • SiteLint does not report autocomplete="off" on a user-information field where a valid input-purpose token should be provided.
  • SiteLint does not report an obvious mismatch where a ZIP or postal code field uses autocomplete="street-address".

Recommendation: Broaden 1.3.5 coverage beyond token validation to include missing, disabled, or mismatched autocomplete values on common user-information fields.
16 AAA 1.3.6 Identify Purpose
  • Skipped: This success criterion is about whether the purpose of controls, icons, and regions can be programmatically identified for personalization. Basic accessible-name problems are covered elsewhere, but meaningful 1.3.6 testing requires a defined semantic purpose taxonomy and judgment about whether each component's purpose is exposed in a way personalization tools can use.
17 A 1.4.1 Use of Color
  • Caution: This success criterion is only partly testable through automation. SiteLint's current rule can detect a narrow link-specific pattern where links are visually indistinguishable from surrounding text, but it does not fully evaluate all uses of color.
  • SiteLint correctly reports the failing examples where non-empty links inherit the surrounding paragraph text color and remove text decoration, making them visually difficult to identify as links.
  • SiteLint does not report the passing examples with an underline, a font-style difference, or button-like styling because those links provide non-color visual cues or otherwise differ from the surrounding text in ways the current rule evaluates.
  • SiteLint's current rule does not fully evaluate other 1.4.1 scenarios, including non-underlined links distinguished only by color, error information conveyed only by color, red/green status indicators, chart legends, or selected states shown only through color.
  • Several broader 1.4.1 scenarios require context, visual comparison, interaction testing, or human judgment to determine whether color is the only cue conveying meaning.

Recommendation: Treat the current rule as useful partial WCAG 1.4.1 coverage for visually indistinguishable links. Consider future heuristics for color-only links, form errors, status indicators, selected states, and chart/data-visualization review candidates, but avoid presenting those as confirmed failures without sufficient context.
18 A 1.4.2 Audio Control
  • Caution: This success criterion is only partly testable through automation. SiteLint's autoplay rule detects explicit autoplay patterns such as autoplay attributes and iframe URLs containing autoplay=1, but it does not fully determine audibility, duration, playback behavior, or whether user controls satisfy the requirement.
  • SiteLint reports the expected autoplay examples, including audio autoplay, video autoplay, iframe src values containing autoplay=1, and iframe allow values containing autoplay=1.
  • SiteLint does not report the examples without autoplay declarations.
  • SiteLint also reports muted autoplay video and autoplaying audio with controls because the current rule flags the autoplay pattern itself. These reports need nuance because muted media and available controls affect the final WCAG 1.4.2 judgment.
  • Secondary media findings, such as captions or descriptions, are acceptable on this page because the same media element can legitimately trigger separate WCAG issues. They should be interpreted separately from the autoplay findings.
  • Reported media elements did not reliably highlight on the page. This may be affected by placeholder media paths and about:blank iframes, but SiteLint should ideally highlight rendered media elements or provide a fallback selector, DOM path, or source snippet.

Recommendation: Treat the autoplay rule as useful partial WCAG 1.4.2 coverage. Consider future improvements for muted autoplay handling, media duration checks, iframe parameter coverage, visual highlighting or fallback location messaging, and a separate real-media test page for playback and highlighting behavior.
19 AA 1.4.3 Contrast (Minimum)
  • Caution: SiteLint provides useful automated WCAG 1.4.3 coverage for many ordinary HTML text contrast cases, but the rule does not fully evaluate every rendered contrast scenario.
  • SiteLint correctly reports several intended text contrast failures on the page, including normal text, large text, button text, and link text with insufficient contrast.
  • SiteLint incorrectly reports the decorative low-contrast divider characters as a confirmed 1.4.3 issue, even though those characters are marked aria-hidden="true" and are not needed to understand the content.
  • SiteLint did not report the low-contrast placeholder text example, even though placeholder text can communicate instructions or examples and should meet the applicable contrast threshold when it conveys information.
  • Additional potentially missed cases include text over gradients or images, text affected by ancestor opacity, and visible SVG <text>.
  • If the WCAG AAA filter is enabled, SiteLint may also report WCAG 1.4.6 Contrast (Enhanced) on this page. That is expected because text that fails 1.4.3 AA will also fail the stricter AAA threshold.

Recommendation: Keep 1.4.3 as an automated contrast rule, but refine it so decorative aria-hidden text is not reported as a confirmed failure, add placeholder contrast support, and consider rendered-background, opacity/compositing, and SVG text contrast enhancements.
20 AA 1.4.4 Resize Text
  • Caution: This success criterion is only partly testable through automation. SiteLint's current zoom-disabled rule detects one narrow pattern: viewport metadata that disables or restricts browser zoom.
  • SiteLint should report the active viewport meta tag in the document head because it includes user-scalable=no.
  • The reported viewport meta element cannot be visually highlighted on the page because it is a non-rendered head element. SiteLint should rely on the DOM path, source snippet, or DevTools location for these findings.
  • SiteLint's current rule does not evaluate whether text can actually be resized to 200 percent without clipping, overlap, or loss of functionality.
  • SiteLint's current rule does not detect layout problems after zoom, such as fixed-height containers clipping text, absolutely positioned text overlapping, very small fixed-size text, or controls becoming unusable after resizing.

Recommendation: Treat zoom-disabled as useful partial WCAG 1.4.4 coverage for viewport zoom restrictions. Consider a separate rendered-page/runtime test that evaluates layout at 200 percent zoom or increased text size, checks for clipping and overlap, and verifies that controls remain operable.
21 AA 1.4.5 Images of Text
  • Skipped: This requires inspecting image content to determine whether the image contains text, whether that text is essential, and whether real text could be used instead. The HTML can expose an image element, but it does not reliably reveal the visual text inside the image.
22 AAA 1.4.6 Contrast (Enhanced)
  • Caution: SiteLint provides useful automated WCAG 1.4.6 coverage for ordinary HTML text contrast cases, especially examples that pass WCAG 1.4.3 AA but fail the stricter WCAG 1.4.6 AAA thresholds.
  • SiteLint correctly reports the core enhanced-contrast failures that distinguish WCAG 1.4.6 from WCAG 1.4.3, including normal text, large text, button text, and link text examples that pass AA but fail AAA.
  • SiteLint incorrectly reports the decorative low-contrast divider characters as a confirmed 1.4.6 issue, even though those characters are marked aria-hidden="true" and are not needed to understand the content.
  • SiteLint did not report the low-contrast placeholder text example, even though placeholder text can communicate instructions or examples and should meet the applicable enhanced contrast threshold when it conveys information.
  • SiteLint did not report the gradient background, opacity/compositing, or visible SVG <text> examples. These should be treated as rendered-analysis, rule-refinement, or rule-expansion areas.
  • The 1.4.6 limitations largely mirror the confirmed 1.4.3 limitations, but the 1.4.6 page now confirms those behaviors under the enhanced-contrast rule as well.

Recommendation: Keep 1.4.6 as a separate enhanced-contrast rule using the stricter 7:1 threshold for normal text and 4.5:1 threshold for large-scale text. Prioritize decorative aria-hidden false positives, placeholder text contrast, rendered-background analysis, opacity/compositing, and SVG <text> contrast as rule-refinement or expansion areas.
23 AAA 1.4.7 Low or No Background Audio
  • Skipped: This requires listening to prerecorded audio to determine whether background sounds are absent, low enough, or controllable relative to speech. The required judgment depends on the actual audio track, not the presence of an audio element in markup.
24 AAA 1.4.8 Visual Presentation
  • Skipped: This requires determining whether users can control visual presentation details such as colors, line width, text alignment, line spacing, and text size. A useful test would need to change those presentation settings or verify that the page supports them, then inspect whether the content remains usable.
25 AAA 1.4.9 Images of Text (No Exception)
  • Skipped: This requires inspecting the image itself to determine whether it contains text and whether that text is essential or purely decorative. The markup alone cannot reliably identify text embedded inside images or decide whether an exception applies.
26 AA 1.4.10 Reflow
  • Skipped: This requires rendering the page at a narrow viewport, commonly 320 CSS pixels wide, and checking whether content can reflow without two-dimensional scrolling. The result depends on the active viewport size and rendered layout, so the test runner would need to control or simulate that viewport.
27 AA 1.4.11 Non-text Contrast
  • SiteLint does not currently report any of the intentionally failing non-text contrast examples on the page.
  • SiteLint does not report low-contrast form control boundaries, including an input border and button border with insufficient contrast against the adjacent background.
  • SiteLint does not report a low-contrast focus indicator defined in CSS.
  • SiteLint does not report low-contrast meaningful graphical objects, including an SVG icon and a status indicator.

Recommendation: Add WCAG 1.4.11 coverage for common machine-detectable non-text contrast patterns, starting with control boundaries and focus indicators, then expanding to meaningful SVG icons and status indicators where context supports evaluation.
28 AA 1.4.12 Text Spacing
  • Skipped: This requires applying specific text-spacing overrides for line height, paragraph spacing, letter spacing, and word spacing, then checking for clipping, overlap, or loss of content/functionality. The test depends on rendering the page after those overrides are applied.
29 AA 1.4.13 Content on Hover or Focus
  • Skipped: This requires triggering hover and keyboard focus states, then checking whether the revealed content can be dismissed, hovered, and kept visible long enough to use. A meaningful test would need to move the pointer, tab to controls, press Escape, and observe the resulting behavior.
30 A 2.1.1 Keyboard
  • Skipped: This requires operating every interactive feature with the keyboard and confirming the same functionality is available without a mouse. The HTML can suggest whether controls are focusable, but a human or interaction test must actually tab, activate controls, and verify behavior.
31 A 2.1.2 No Keyboard Trap
  • Skipped: This requires moving keyboard focus into and out of components to confirm the user is not trapped. The key issue is runtime focus behavior, including modals, widgets, embedded content, and scripts that may intercept keyboard input.
  • SiteLint purports to provide WCAG 2.1.2 coverage, but the current rule no-keyboard-trap does not actually test for keyboard traps. It selects the body element and reports a page-level manual-verification prompt. It does not move focus, test Tab or Shift+Tab behavior, enter or exit components, inspect modals/widgets/iframes, or determine whether focus can escape.
32 AAA 2.1.3 Keyboard (No Exception)
  • Skipped: This requires testing all functionality with a keyboard and does not allow the usual path-dependent exception. Evaluation requires activating controls and workflows directly, because markup alone cannot prove that every scripted function is keyboard-operable.
33 A 2.1.4 Character Key Shortcuts
  • Skipped: This requires identifying character-key shortcuts and testing whether they can be turned off, remapped, or limited to focused components. The issue is usually implemented through JavaScript keyboard event behavior, so it must be exercised interactively.
34 A 2.2.1 Timing Adjustable
  • Caution: This success criterion is only partly testable through automation. SiteLint's current no-meta-http-equiv-refresh rule detects one narrow timing pattern: pages that use meta http-equiv="refresh" to refresh or redirect after a time interval.
  • SiteLint correctly reports the two intentional meta refresh examples in the document head: one page refresh and one timed redirect.
  • The reported meta elements cannot be visually highlighted on the page because they are non-rendered head elements. SiteLint should rely on the DOM path, source snippet, or DevTools location for these findings.
  • SiteLint's current rule does not evaluate JavaScript-based timeouts, session-expiration workflows, timer widgets, or auto-updating content that occurs without a meta refresh.
  • Complete 2.2.1 evaluation usually requires runtime observation, interaction testing, and judgment about whether users can turn off, adjust, or extend the time limit, and whether any WCAG exception applies.

Recommendation: Treat no-meta-http-equiv-refresh as useful partial WCAG 2.2.1 coverage. Consider normalizing http-equiv values case-insensitively and adding a UI fallback message for non-rendered head elements that cannot be visually highlighted.
35 A 2.2.2 Pause, Stop, Hide
  • Caution: This success criterion is only partly testable through automation. SiteLint's animation.ts can detect long-running or infinite CSS animations, but it does not fully evaluate whether users have a working pause, stop, or hide mechanism.
  • SiteLint should report the CSS animation examples with durations longer than five seconds or infinite iteration counts.
  • SiteLint should not report the short, paused, zero-iteration, or zero-duration animation examples.
  • SiteLint correctly reports the infinite scrolling text scenario because it uses an infinite CSS animation with a duration longer than five seconds.
  • SiteLint does not fully evaluate auto-updating content, JavaScript-driven visual changes, user-triggered animation, or whether pause/resume controls actually operate the moving content.

Recommendation: Treat animation.ts as useful partial WCAG 2.2.2 coverage. Consider future runtime checks for pause, stop, or hide controls and a separate rule for auto-updating content, while recognizing that some cases still require human judgment.
36 AAA 2.2.3 No Timing
  • Skipped: This requires understanding whether time limits are part of the activity and whether timing is essential. That usually depends on the task, workflow, and user interaction, not just the presence of timers in code.
37 AAA 2.2.4 Interruptions
  • Skipped: This requires triggering or waiting for interruptions such as alerts, popups, or forced context changes, then determining whether users can postpone or suppress them. The issue depends on behavior that occurs after page load or during a workflow.
38 AAA 2.2.5 Re-authenticating
  • Skipped: This requires testing a real or simulated authenticated session, letting it expire, re-authenticating, and confirming whether the user can continue without data loss. That cannot be determined from a standalone page's markup.
39 AAA 2.2.6 Timeouts
  • Skipped: This requires identifying inactivity timeouts that could cause data loss and confirming whether users are warned about the duration. Testing requires a running workflow with timeout behavior, not just static code inspection.
40 A 2.3.1 Three Flashes or Below Threshold
  • Skipped: This requires observing flashing content over time and measuring whether flashes exceed the allowed threshold. The relevant information is frequency, area, and timing of visual changes, which are not available from a single static DOM snapshot.
41 AAA 2.3.2 Three Flashes
  • Skipped: This requires measuring flashing behavior over time and confirming that no content flashes more than three times in one second. A test must observe animation/video frames or scripted visual changes, not just markup.
42 AAA 2.3.3 Animation from Interactions
  • Skipped: This requires performing user interactions that trigger motion animation and checking whether non-essential animation can be disabled. The test depends on interaction-triggered behavior and user preferences such as reduced motion.
43 A 2.4.1 Bypass Blocks
  • Caution: SiteLint's current bypass-blocks rule reports a manual-verification item when it detects visible navigation-like structures such as nav, header, [aria-label="Skip links"], .main-navigation, or .skip-links.
  • SiteLint reports the page when a real nav element is present. This confirms that the rule can trigger, but it does not confirm automated WCAG 2.4.1 pass/fail detection.
  • The current rule does not actually determine whether the page passes or fails WCAG 2.4.1. It does not check whether a valid skip link exists, whether the skip link target exists, whether the skip link becomes visible on focus, whether the page has a main landmark, or whether the available bypass mechanism is sufficient.
  • The broken skip-link-target example on the test page is presented as a code sample because an actual broken fragment link triggers SiteLint's separate same-page-link rule under WCAG 2.4.4, not the bypass-blocks rule.

Recommendation: Keep bypass-blocks as a manual-verification prompt unless SiteLint adds concrete checks for missing or broken bypass mechanisms. Confirmed 2.4.1 findings should identify specific issues such as missing main landmark, missing skip link on pages with repeated navigation, missing skip-link target, skip link not visible on focus, or a bypass link that does not move the user to the intended content.
44 A 2.4.2 Page Titled
  • Caution: SiteLint's current unclear-page-title rule provides useful partial WCAG 2.4.2 coverage by checking the active document title for missing, empty, whitespace-only, and several known generic/default title strings.
  • SiteLint correctly reports the active page title Untitled Document under WCAG 2.4.2 because it does not identify the content or purpose of the page.
  • The reported title element cannot be visually highlighted on the page because it is a non-rendered head element. SiteLint's fallback message for non-rendered elements is appropriate here.
  • The current rule does not fully evaluate whether a title describes the page topic or purpose. It may not catch vague but common titles such as Home, Page, Welcome, Dashboard, or short organization-only titles such as APAP.
  • The current single-page rule does not evaluate duplicate titles across a site, whether the title matches the visible page topic, or whether a single-page application updates the document title after route changes.
  • The iframe-title issue is separate from WCAG 2.4.2 and maps more appropriately to WCAG 4.1.2 because it concerns the iframe element's accessible name, not the active document title.

Recommendation: Keep unclear-page-title as automated WCAG 2.4.2 coverage for objective title failures. Consider adding review-candidate logic for likely-generic titles, site-wide duplicate-title analysis, and single-page-application route-change checks.
45 A 2.4.3 Focus Order
  • SiteLint appropriately reports the positive tabindex example and the redundant tabindex="0" example under WCAG 2.4.3.
  • SiteLint also uses focus-order-preserved as a manual-verification trigger, but it does not automatically evaluate rendered focus order.
  • The visual-order mismatch remains the key gap: SiteLint should compare keyboard focus sequence with visual order, reading order, and workflow order where possible.

Recommendation: Keep positive-tabindex and misused-tabindex-attribute as concrete partial 2.4.3 checks. Treat focus-order-preserved as a manual prompt unless rendered focus-order analysis is added.
46 A 2.4.4 Link Purpose (In Context)
  • Caution: SiteLint groups several link-purpose and link-name rules under WCAG 2.4.4, including aria-labelledby-association, aria-labelledby-association-empty-element, empty-link-element, img-alt-duplicate-text-link, link-purpose-in-context, and links-same-content-different-url.
  • SiteLint correctly flags several failing examples on the page: missing aria-labelledby reference, empty aria-labelledby reference, empty-link content, duplicate image/link text, same-page link with missing target, and same-content/different-URL links when manual-verification findings are enabled.
  • SiteLint does not report secondary examples 3, 4, or 9 as live WCAG 2.4.4 findings. These include generic link text, repeated generic “Read more” links, and an icon-only link with no accessible name. SiteLint rules should be improved where these examples represent real link-purpose failures that the current rules miss.
  • links-same-content-different-url is a useful concrete pattern-based review finding, but it only appears when manual-verification findings are enabled. The report says it found 3 links with the same text Register pointing to different URLs, but it does not list the affected links, destination URLs, selectors, XPaths, or page highlight buttons.
  • link-purpose-in-context appears to be a broad manual-verification prompt that detects the presence of links and asks for human review. It should not be treated as evidence of automated WCAG 2.4.4 pass/fail coverage.
  • The empty-link-element finding is reported in the audit UI as WCAG 2.4.6 “Links should have meaningful, descriptive content,” even though the source file appears under wcag/2.4.4/H30/empty-link-element.ts. The source-code location and audit UI mapping should be reconciled.
  • The same-page-link missing-target rule is reported in the audit UI as WCAG 2.4.4, but the Bitbucket rule file appears to live under a WCAG 2.1.1 path. The source-code categorization and UI categorization should be verified.

Recommendation: Distinguish concrete automated findings, concrete pattern-based manual findings, and broad manual review prompts. Pattern-based findings such as links-same-content-different-url should include affected links, destination URLs, selectors or XPaths, and highlight options when possible. Broad prompts such as link-purpose-in-context should remain clearly labeled as manual verification.
47 AA 2.4.5 Multiple Ways
  • Skipped: This success criterion applies across a set of web pages. Testing requires reviewing site navigation, search, sitemap, related links, or other page-finding mechanisms across the site, not a single isolated HTML page.
48 AA 2.4.6 Headings and Labels
  • SiteLint currently reports only failing example 2 as a WCAG 2.4.6 issue: “Heading element exists but has no content.”
  • SiteLint does not report failing example 1, failing example 3, or failing example 5 as WCAG 2.4.6 findings, even though those examples represent headings or labels that are present but not descriptive enough.

Recommendation: Treat empty headings as confirmed automated WCAG 2.4.6 findings. Improve SiteLint rules to catch or review headings and labels that are present but not descriptive enough, including generic headings, vague form labels, and repeated generic button labels. Keep 2.4.6 label/heading clarity separate from related link-purpose findings under WCAG 2.4.4 or 2.4.9.
49 AA 2.4.7 Focus Visible
  • SiteLint uses focus-visible as a manual-verification trigger for WCAG 2.4.7.
  • The current trigger does not evaluate whether focus indicators are actually visible, including removed, transparent, or visually imperceptible focus states.

Recommendation: Keep focus-visible as a manual prompt unless SiteLint adds rendered focus-state testing that moves focus through controls and inspects the visible indicator.
50 AAA 2.4.8 Location
  • Skipped: This success criterion applies across a set of web pages and requires determining whether users can identify their location within the site or process. Testing requires reviewing navigation state, breadcrumbs, page hierarchy, or other location cues across multiple pages.
51 AAA 2.4.9 Link Purpose (Link Only)
  • SiteLint caught failing examples 1, 2, 3, and 4 as WCAG 2.4.9 link-purpose issues.
  • SiteLint did not report failing example 5, where the link text is a bare URL with an ambiguous path: https://example.com/x/93821.
  • The missed bare-URL example may be intentional if SiteLint treats visible URL text as sufficiently descriptive by default, but the example demonstrates a link-purpose risk because the URL path does not identify the destination or purpose in human-readable terms.
  • WCAG 2.4.9 is stricter than WCAG 2.4.4: link purpose generally needs to be clear from the link text alone, not only from surrounding or programmatically determinable context.

Recommendation: Treat generic link text such as click here, learn more, read more, and details as strong WCAG 2.4.9 review candidates. Also consider bare URL text with ambiguous paths as a review candidate when the URL does not communicate a human-readable purpose. When repeated generic links are reported, include each affected link, destination URL, selector or XPath, and a highlight option when possible.
52 AAA 2.4.10 Section Headings
  • Caution: The 2.4.10 test page intentionally violates the standard test-page template by using visually styled div elements instead of native headings. This is necessary because the no_headings rule only fires when the document contains zero h1 through h6 elements.
  • SiteLint's no_headings rule checks only whether native heading elements exist anywhere on the page. If at least one native heading is present, the rule returns without reporting.
  • no_headings should be treated as a review candidate, not a confirmed WCAG 2.4.10 failure. A page with no native headings may indicate poor section organization, but WCAG 2.4.10 still requires judgment about whether the page has sections that need headings and whether those headings organize the content effectively.
  • no_headings alone is not complete WCAG 2.4.10 coverage. It does not evaluate whether section headings are needed, whether existing headings are useful, whether visual section labels should be semantic headings, whether heading text is descriptive, or whether the content is organized effectively.
