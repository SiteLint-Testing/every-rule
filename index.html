WCAG Test Page Index

This index tracks the WCAG success-criterion test pages being created to evaluate SiteLint coverage. Each linked page contains representative passing and failing scenarios for that success criterion, followed by observations and recommendations when SiteLint behavior has been reviewed.

Created (10) and skipped (61) WCAG test pages
# SC Title Notes
1 1.1.1 Non-text Content
  • No SiteLint issues requiring follow-up identified from the current first-pass test page.
2 1.2.1 Audio-only and Video-only (Prerecorded)
  • Caution: This success criterion is only partly testable through static markup. Whether the provided alternative is actually equivalent to the prerecorded audio-only or video-only content requires human review of the media and its alternative.
  • SiteLint incorrectly flags a passing video-only example that provides an associated text alternative, even though WCAG 1.2.1 allows prerecorded video-only content to pass through an alternative for time-based media.
  • The current media rule appears to treat the absence of a track element or audio source as sufficient evidence of failure, which is too narrow for 1.2.1.

Recommendation: 1.2.1 logic should account for the valid text-alternative pathway rather than treating the absence of embedded track/audio markup as conclusive.
3 1.2.2 Captions (Prerecorded)
  • Caution: A static page can test whether a captions mechanism appears to be present, but it cannot verify whether the captions are accurate, complete, synchronized, or include relevant non-speech audio.
  • SiteLint incorrectly flags a passing example that is clearly labeled as a media alternative for text, which is an exception under WCAG 1.2.2.
  • SiteLint catches the missing-captions failing scenario only through a finding mapped to 1.2.1, not 1.2.2.
  • SiteLint misses the subtitles-only failing scenario, suggesting that subtitles may be treated as equivalent to captions.

Recommendation: SiteLint needs explicit, accurately mapped 1.2.2 coverage and should not treat kind="subtitles" as a guaranteed substitute for captions.
4 1.2.3 Audio Description or Media Alternative (Prerecorded)
  • Caution: This success criterion is only partly testable through static markup. Whether audio description or a media alternative fully conveys the necessary visual and auditory information requires human review of the media content.
  • SiteLint incorrectly flags a passing example that provides a full media alternative in text form.
  • SiteLint incorrectly flags a passing example that is clearly labeled as a media alternative for text.
  • SiteLint flags the genuine failing scenario, but reports it under WCAG 1.2.1 rather than 1.2.3.

Recommendation: SiteLint needs explicit 1.2.3 coverage that accounts for audio description, full media alternatives, and the media-alternative-for-text exception.
5 1.2.4 Captions (Live)
  • Skipped: This success criterion applies to live synchronized media. A meaningful test requires real-time or simulated-live audio/video and a judgment about whether captions are provided as the live content occurs. Reviewing the static HTML only cannot determine whether live captions are present, accurate, synchronized, or complete.
6 1.2.5 Audio Description (Prerecorded)
  • Caution: This success criterion is difficult to evaluate through automation because the need for audio description depends on whether important visual information is already conveyed in the existing audio track.
  • SiteLint incorrectly flags a passing example where important visual information is already conveyed in the main audio track, a valid 1.2.5 scenario.
  • SiteLint flags the genuine failing scenario, but again reports it under WCAG 1.2.1 rather than 1.2.5.

Recommendation: SiteLint should not treat the absence of a descriptions track as a confirmed 1.2.5 failure. This often requires focused manual verification.
7 1.2.6 Sign Language (Prerecorded)
  • Skipped: This success criterion requires reviewing the prerecorded video content to determine whether a sign language interpretation is actually provided and whether it covers the spoken audio. HTML can show that a video exists, but it cannot confirm that sign language interpretation appears in the media.
8 1.2.7 Extended Audio Description (Prerecorded)
  • Skipped: This success criterion requires reviewing the timing and content of the prerecorded media to determine whether ordinary audio description cannot fit into the available pauses and extended audio description is therefore required. That judgment depends on the media timeline, not just the surrounding markup.
9 1.2.8 Media Alternative (Prerecorded)
  • Caution: This success criterion is only partly testable through static markup. A page can show whether a clearly associated media alternative is present or absent, but whether that alternative fully captures the important visual and auditory information requires human review of the media and the alternative.
  • SiteLint incorrectly flags both passing examples that provide full media alternatives: one for prerecorded synchronized media and one for prerecorded video-only media.
  • SiteLint flags the two genuine failing scenarios, but reports them under WCAG 1.2.1 rather than WCAG 1.2.8.

Recommendation: SiteLint needs explicit, accurately mapped 1.2.8 coverage that accounts for full media alternatives rather than reusing the generic “no track or audio source” media heuristic.
10 1.2.9 Audio-only (Live)
  • Skipped: This success criterion applies to live audio-only content. A meaningful test requires live or simulated-live audio and a judgment about whether an equivalent real-time text alternative is available. Static HTML cannot confirm the live audio content or the accuracy and timing of its alternative.
11 1.3.1 Info and Relationships
  • SiteLint correctly flags the ungrouped radio-button example under WCAG 1.3.1, but the rule message says “Checkboxes should be grouped together” even though the controls are radio buttons.
  • SiteLint correctly flags the data table with visual headers but no programmatic header cells under WCAG 1.3.1.
  • SiteLint flags the visible-but-unassociated form label, but maps the finding to WCAG 3.3.2 rather than 1.3.1.
  • SiteLint also reports a WCAG 2.5.8 target-size issue on the radio buttons, which is unrelated to this 1.3.1 test page and may need to be avoided through test-page styling.

Recommendation: Update the grouped-control rule message to include radio buttons, consider whether form-label relationship failures should also map to 1.3.1, and keep the page focused on machine-detectable structural relationship failures.
12 1.3.2 Meaningful Sequence
  • Caution: This success criterion is only partly testable through automation. Automated tests can detect some risky patterns where CSS changes visual order, but whether the changed sequence affects meaning may still require human review.
  • SiteLint does not currently report either failing example on the page, even though both use explicit CSS order values that make the visual order differ from the DOM order.

Recommendation: SiteLint should consider a targeted WCAG 1.3.2 manual-verification rule for flex/grid reordering patterns where source order and visual order may diverge.
13 1.3.3 Sensory Characteristics
  • Skipped: This success criterion requires deciding whether instructions rely only on sensory characteristics such as shape, color, size, visual location, orientation, or sound. A tool can search for risky words like “red,” “left,” or “round,” but a human still has to read the surrounding content and inspect the referenced control to determine whether a non-sensory identifier is also provided.
14 1.3.4 Orientation
  • Skipped: This success criterion requires checking whether the page can be viewed and operated in both portrait and landscape orientation. A meaningful test would need to render or simulate the page in both orientations and determine whether content or functionality is blocked in one of them.
15 1.3.5 Identify Input Purpose
  • SiteLint correctly reports an invalid autocomplete token under WCAG 1.3.5.
  • SiteLint does not report missing autocomplete tokens on common user-information fields, such as full name.
  • SiteLint does not report autocomplete="off" on a user-information field where a valid input-purpose token should be provided.
  • SiteLint does not report an obvious mismatch where a ZIP or postal code field uses autocomplete="street-address".

Recommendation: Broaden 1.3.5 coverage beyond token validation to include missing, disabled, or mismatched autocomplete values on common user-information fields.
16 1.3.6 Identify Purpose
  • Skipped: This success criterion is about whether the purpose of controls, icons, and regions can be programmatically identified for personalization. Basic accessible-name problems are covered elsewhere, but meaningful 1.3.6 testing requires a defined semantic purpose taxonomy and judgment about whether each component's purpose is exposed in a way personalization tools can use.
17 1.4.1 Use of Color
  • Skipped: This success criterion requires deciding whether color is the only cue conveying meaning. A tool can detect red text, green status dots, or color changes, but a human must inspect the surrounding content to determine whether there is also text, an icon, a pattern, a label, or another non-color cue that communicates the same information.
18 1.4.2 Audio Control
  • Skipped: This success criterion requires observing whether audio starts automatically, continues for more than three seconds, and can be paused, stopped, or controlled independently of system volume. That requires running the page and testing playback behavior, not just inspecting the audio markup.
19 1.4.3 Contrast (Minimum)
20 1.4.4 Resize Text
  • Skipped: This success criterion requires increasing text size up to 200 percent and then checking whether content remains readable and functional without loss of information. The test depends on applying a zoom or text-resize condition and reviewing the rendered result, not just reading the original HTML and CSS.
21 1.4.5 Images of Text
  • Skipped: This success criterion requires inspecting image content to determine whether the image contains text, whether that text is essential, and whether real text could be used instead. The HTML can expose an image element, but it does not reliably reveal the visual text inside the image.
22 1.4.6 Contrast (Enhanced)
23 1.4.7 Low or No Background Audio
  • Skipped: This success criterion requires listening to prerecorded audio to determine whether background sounds are absent, low enough, or controllable relative to speech. The required judgment depends on the actual audio track, not the presence of an audio element in markup.
24 1.4.8 Visual Presentation
  • Skipped: This success criterion requires determining whether users can control visual presentation details such as colors, line width, text alignment, line spacing, and text size. A useful test would need to change those presentation settings or verify that the page supports them, then inspect whether the content remains usable.
25 1.4.9 Images of Text (No Exception)
  • Skipped: This success criterion requires inspecting the image itself to determine whether it contains text and whether that text is essential or purely decorative. The markup alone cannot reliably identify text embedded inside images or decide whether an exception applies.
26 1.4.10 Reflow
  • Skipped: This success criterion requires rendering the page at a narrow viewport, commonly 320 CSS pixels wide, and checking whether content can reflow without two-dimensional scrolling. The result depends on the active viewport size and rendered layout, so the test runner would need to control or simulate that viewport.
27 1.4.11 Non-text Contrast
  • SiteLint does not currently report any of the intentionally failing non-text contrast examples on the page.
  • SiteLint does not report low-contrast form control boundaries, including an input border and button border with insufficient contrast against the adjacent background.
  • SiteLint does not report a low-contrast focus indicator defined in CSS.
  • SiteLint does not report low-contrast meaningful graphical objects, including an SVG icon and a status indicator.

Recommendation: Add WCAG 1.4.11 coverage for common machine-detectable non-text contrast patterns, starting with control boundaries and focus indicators, then expanding to meaningful SVG icons and status indicators where context supports evaluation.
28 1.4.12 Text Spacing
  • Skipped: This success criterion requires applying specific text-spacing overrides for line height, paragraph spacing, letter spacing, and word spacing, then checking for clipping, overlap, or loss of content/functionality. The test depends on rendering the page after those overrides are applied.
29 1.4.13 Content on Hover or Focus
  • Skipped: This success criterion requires triggering hover and keyboard focus states, then checking whether the revealed content can be dismissed, hovered, and kept visible long enough to use. A meaningful test would need to move the pointer, tab to controls, press Escape, and observe the resulting behavior.
30 2.1.1 Keyboard
  • Skipped: This success criterion requires operating every interactive feature with the keyboard and confirming the same functionality is available without a mouse. The HTML can suggest whether controls are focusable, but a human or interaction test must actually tab, activate controls, and verify behavior.
31 2.1.2 No Keyboard Trap
  • Skipped: This success criterion requires moving keyboard focus into and out of components to confirm the user is not trapped. The key issue is runtime focus behavior, including modals, widgets, embedded content, and scripts that may intercept keyboard input.
32 2.1.3 Keyboard (No Exception)
  • Skipped: This success criterion requires testing all functionality with a keyboard and does not allow the usual path-dependent exception. Evaluation requires activating controls and workflows directly, because markup alone cannot prove that every scripted function is keyboard-operable.
33 2.1.4 Character Key Shortcuts
  • Skipped: This success criterion requires identifying character-key shortcuts and testing whether they can be turned off, remapped, or limited to focused components. The issue is usually implemented through JavaScript keyboard event behavior, so it must be exercised interactively.
34 2.2.1 Timing Adjustable
  • Skipped: This success criterion requires identifying time limits and testing whether the user can turn them off, adjust them, or extend them before expiration. The evaluation depends on workflow timing, warnings, and available controls during runtime.
35 2.2.2 Pause, Stop, Hide
  • Skipped: This success criterion requires observing content that moves, blinks, scrolls, or auto-updates for long enough to determine whether it starts automatically and whether pause, stop, or hide controls work. The relevant behavior occurs over time after the page loads.
36 2.2.3 No Timing
  • Skipped: This success criterion requires understanding whether time limits are part of the activity and whether timing is essential. That usually depends on the task, workflow, and user interaction, not just the presence of timers in code.
37 2.2.4 Interruptions
  • Skipped: This success criterion requires triggering or waiting for interruptions such as alerts, popups, or forced context changes, then determining whether users can postpone or suppress them. The issue depends on behavior that occurs after page load or during a workflow.
38 2.2.5 Re-authenticating
  • Skipped: This success criterion requires testing a real or simulated authenticated session, letting it expire, re-authenticating, and confirming whether the user can continue without data loss. That cannot be determined from a standalone page's markup.
39 2.2.6 Timeouts
  • Skipped: This success criterion requires identifying inactivity timeouts that could cause data loss and confirming whether users are warned about the duration. Testing requires a running workflow with timeout behavior, not just static code inspection.
40 2.3.1 Three Flashes or Below Threshold
  • Skipped: This success criterion requires observing flashing content over time and measuring whether flashes exceed the allowed threshold. The relevant information is frequency, area, and timing of visual changes, which are not available from a single static DOM snapshot.
41 2.3.2 Three Flashes
  • Skipped: This success criterion requires measuring flashing behavior over time and confirming that no content flashes more than three times in one second. A test must observe animation/video frames or scripted visual changes, not just markup.
42 2.3.3 Animation from Interactions
  • Skipped: This success criterion requires performing user interactions that trigger motion animation and checking whether non-essential animation can be disabled. The test depends on interaction-triggered behavior and user preferences such as reduced motion.
