WCAG 1.3.5: Identify Input Purpose

View main list

Brief summary: Input fields that collect information about the user should expose their purpose programmatically when the purpose is included in the WCAG-defined list of input purposes. In HTML, this is commonly done with valid autocomplete tokens such as name, email, tel, street-address, and postal-code.

View the W3C Understanding page for WCAG 1.3.5 Identify Input Purpose.

These examples focus on user-information fields where an automated accessibility tool can reasonably inspect whether a valid and appropriate autocomplete token is present. The page intentionally avoids fields that do not collect information about the user, because those are outside the intended scope of this success criterion.

Passing examples

Passing example 1: Full name field has a valid input-purpose token

This succeeds because the field collects the user's full name and uses autocomplete="name" to expose that purpose programmatically.

Passing example 2: Email field has a valid input-purpose token

This succeeds because the field collects the user's email address and uses autocomplete="email".

Passing example 3: Telephone field has a valid input-purpose token

This succeeds because the field collects the user's telephone number and uses autocomplete="tel".

Passing example 4: Address fields have valid input-purpose tokens

This succeeds because the address-related fields use valid tokens that identify the specific user information being requested.

Failing examples

Failing example 1: Full name field is missing an input-purpose token

This fails because the field collects the user's full name, but it does not provide an autocomplete token to identify the input purpose programmatically.

SiteLint observation: SiteLint does not report this intentionally failing example. The field collects the user's full name, but no autocomplete token is present.

Recommendation: SiteLint should consider detecting missing autocomplete tokens on common user-information fields, such as name, email, telephone, and address fields. This may require label/name/id heuristics, but it is central to meaningful WCAG 1.3.5 coverage.

Failing example 2: Email field uses an invalid autocomplete value

This fails because the field collects the user's email address, but the autocomplete value is not a valid token for identifying the input purpose.

SiteLint observation: SiteLint correctly reports this failing example under WCAG 1.3.5 because autocomplete="email-address" is not a valid autocomplete token.

Recommendation: Keep this as a strong automated 1.3.5 test scenario. Invalid autocomplete tokens are directly machine-detectable and should be reported confidently.

Failing example 3: Phone field disables autocomplete instead of identifying the purpose

This fails because the field collects the user's phone number, but it uses autocomplete="off" instead of a valid input-purpose token such as tel.

SiteLint observation: SiteLint does not report this intentionally failing example. The field collects the user's phone number, but autocomplete="off" does not identify the input purpose.

Recommendation: SiteLint should not treat autocomplete="off" as satisfying WCAG 1.3.5 for user-information fields that have a defined autocomplete purpose. A field that asks for a phone number should expose that purpose with a token such as tel.

Failing example 4: Postal code field uses the wrong token

This fails because the field asks for the user's ZIP or postal code, but the token identifies a different input purpose.

SiteLint observation: SiteLint does not report this intentionally failing example. The field asks for a ZIP or postal code but uses autocomplete="street-address", which identifies a different input purpose.

Recommendation: SiteLint should consider detecting likely mismatches between common field labels and autocomplete tokens. This may need heuristic confidence scoring, but obvious mismatches such as postal code fields using street-address are useful 1.3.5 candidates.

Current 1.3.5 coverage takeaway

Recommendation: SiteLint currently appears to detect invalid autocomplete tokens, but it does not detect missing tokens, autocomplete="off" on user-information fields, or likely mismatches between field purpose and autocomplete value. This means the current 1.3.5 coverage is useful but too narrow.

The strongest automated coverage is token validation. Broader 1.3.5 coverage should also inspect common user-information fields and determine whether the expected input purpose is absent, disabled, or mismatched.
