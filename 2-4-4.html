WCAG 2.4.4: Link Purpose (In Context)

View main list

Brief summary: The purpose of each link should be understandable from the link text alone or from its programmatically determinable context, except where the purpose would be ambiguous to users in general.

View the W3C Understanding page for WCAG 2.4.4 Link Purpose (In Context).

SiteLint rules grouped under WCAG 2.4.4

SiteLint groups the following rules under WCAG 2.4.4 Link Purpose (In Context):

  • aria-labelledby-association: The aria-labelledby attribute must reference an existing element.
  • aria-labelledby-association-empty-element: An aria-labelledby attribute must reference an element that contains meaningful content.
  • empty-link-element: Links should have meaningful, descriptive content.
  • img-alt-duplicate-text-link: A link and its image may not have duplicate text content.
  • link-purpose-in-context: Verify link purpose in context.
  • links-same-content-different-url: Avoid link elements that have the same content, but different destination URLs.

What this page is intended to test

SiteLint maps several link-related rules to WCAG 2.4.4. Some are direct link-purpose checks, while others overlap with accessible-name or ARIA-reference quality because those issues can prevent users from determining a link's purpose.

This page tests the broader SiteLint 2.4.4 rule family as exposed in the SiteLint audit UI. Confirmed live findings on this page include missing aria-labelledby references, empty aria-labelledby references, duplicate image/link text, same-page links with missing targets, and empty-link content. With manual-verification findings enabled, SiteLint also reports link-purpose-in-context and links-same-content-different-url.

Recommendation: Manual-verification findings need clearer separation in the UI. link-purpose-in-context is a broad prompt that appears to detect that links exist on the page and asks for human review. It is not effectively determining whether specific links pass or fail WCAG 2.4.4. links-same-content-different-url is more actionable because it detects a concrete pattern, but it is still marked as requiring manual verification and can be buried among lower-value review prompts.

Failing or risky examples aligned to SiteLint's 2.4.4 rule family

Failing example 1: Link uses aria-labelledby but references a missing element

This is intended to trigger SiteLint's aria-labelledby-association rule. The link has an aria-labelledby value that references missing-link-label, but no element with that id exists on the page.

Missing labelledby reference test

Recommendation: SiteLint should report this under WCAG 2.4.4 as “The aria-labelledby attribute must reference an existing element.” The link also includes visible text and an aria-label so the page can test the missing aria-labelledby reference without also triggering the unrelated WCAG 2.4.6 link-content rule.

Failing example 2: Link uses aria-labelledby but references an empty element

This triggers SiteLint's aria-labelledby-association-empty-element rule. The referenced label element exists, but it does not contain meaningful content.

Fallback link text

Observed result: SiteLint reports this under WCAG 2.4.4 as “An aria-labelledby attribute must reference an element that contains meaningful content.”

Secondary example 3: Generic link text

Generic link text such as “click here” can relate to link purpose, but SiteLint currently reports this kind of issue under other link-text rules such as WCAG 2.4.6 or WCAG 2.4.9 rather than the 2.4.4 rule family shown on this page. The example is shown as a code sample to keep the live findings focused on WCAG 2.4.4.

<p>For more details about the annual conference, <a href="https://example.com/conference-details">click here</a>.</p>

Recommendation: SiteLint rules should be improved to catch this failure when generic link text does not provide enough link-purpose information from the link text or its programmatically determinable context.

Secondary example 4: Multiple “read more” links to different destinations

Repeated generic link text can be confusing when each link points to a different destination, but SiteLint currently reports this pattern under its out-of-context link-purpose rule mapped to WCAG 2.4.9. The example is shown as a code sample to keep the live findings focused on WCAG 2.4.4.

<a href="https://example.com/news/accessibility-training">Read more</a>
<a href="https://example.com/news/registration-open">Read more</a>

Recommendation: SiteLint rules should be improved to catch this failure when repeated generic link text points to different destinations and the surrounding context is not sufficient or not exposed clearly enough to establish each link purpose.

Failing example 5: Same link text points to different URLs

This is intended to trigger SiteLint's links-same-content-different-url rule. The three links have the same visible text and accessible name, but each link points to a different canonical URL. The links are direct children of a generic div, not inside article or li elements, because the rule treats links inside those containers as contextual exceptions.

Register Register Register

Observed result: SiteLint reports links-same-content-different-url only when manual-verification findings are included. It provides the message: “Found 3 links with the same text Register (accessible name: Register) pointing to different URLs. Every link should have unique content.”

Recommendation: This is a useful pattern-based finding, but SiteLint should list the affected links, their destination URLs, and their selectors or XPaths. It should also provide highlight buttons for each affected link when possible. Without those details, users must search through the page source to find the repeated links, which makes the finding harder to act on.

Failing example 6: Empty link with whitespace-only content

This is intended to trigger SiteLint's empty-link-element rule. The link has an href and contains only whitespace, with no aria-label, no aria-hidden="true", and no other accessible content to explain the link purpose.

Recommendation: SiteLint reports this as “Links should have meaningful, descriptive content” under WCAG 2.4.6 in the audit UI, but the source file for empty-link-element appears in the repository under wcag/2.4.4/H30/empty-link-element.ts. The source-code location and audit UI mapping should be reconciled.

Failing example 7: Link and image have duplicate text

This triggers SiteLint's img-alt-duplicate-text-link rule. The image alternative text duplicates the adjacent link text, which may create redundant or confusing link output depending on how the link is exposed.

Annual report

Observed result: SiteLint reports this under WCAG 2.4.4 as “A link and its image may not have duplicate text content.”

Observed failing example 8: Same-page link points to a missing target

A same-page link that points to a missing target does not take the user to the intended content. This triggers SiteLint's same-page-link rule, which SiteLint maps to WCAG 2.4.4.

Jump to event details

Recommendation: SiteLint reports this as “Same-page links must have a corresponding target” under WCAG 2.4.4 in the audit UI, but the Bitbucket rule file appears to live under a WCAG 2.1.1 path. The source-code categorization and UI categorization should be verified.

Secondary example 9: Icon-only link has no accessible name

An icon-only link with no accessible label does not communicate its purpose, but this can trigger accessible-name or target-size findings outside the current WCAG 2.4.4 focus. The example is shown as a code sample to avoid unrelated live findings.

<a href="https://example.com/download-file"><span aria-hidden="true">Download</span></a>

Recommendation: SiteLint rules should be improved to catch this failure when an icon-only link has no accessible name, while avoiding unrelated target-size or taxonomy noise.

Observed SiteLint rule family

SiteLint correctly flags several failing examples on this page: aria-labelledby-association for failing example 1, aria-labelledby-association-empty-element for failing example 2, empty-link-element / “Links should have meaningful, descriptive content” for failing example 6, img-alt-duplicate-text-link for failing example 7, same-page link missing target for observed failing example 8, and links-same-content-different-url when manual-verification findings are enabled.

SiteLint does not report secondary examples 3, 4, or 9 as live WCAG 2.4.4 findings. Failing example 5 is reported only when manual-verification findings are enabled. Some of these results may reflect intentional taxonomy choices or manual-verification behavior rather than true rule failures, but SiteLint rules should be improved where these examples represent real link-purpose failures that the current rules miss.

Recommended SiteLint treatment

Recommendation: With manual-verification findings enabled, SiteLint reports both link-purpose-in-context and links-same-content-different-url, but these should not be treated the same way. link-purpose-in-context is a general review prompt that appears because links exist on the page. links-same-content-different-url is a concrete pattern-based review finding because it detects multiple links with the same accessible name but different canonical destinations.

Recommendation: SiteLint's broader WCAG 2.4.4 mapping is defensible for many link-purpose problems, but some findings also overlap with accessible-name, ARIA-reference, image-alternative, or keyboard/navigation quality. The UI should make that relationship clear when a finding is classified under WCAG 2.4.4. Mapping discrepancies should also be checked where the audit UI and source-code location appear to point to different WCAG criteria.

Recommendation: The manual-verification bucket contains both useful pattern-based findings and broad prompts that create noise. links-same-content-different-url is useful because it identifies a specific repeated-link pattern. link-purpose-in-context is much less specific because it does not evaluate individual link text, surrounding context, destination accuracy, or whether the page actually satisfies WCAG 2.4.4.

Recommendation: Clarify whether each finding is a direct link-purpose failure, an accessible-name failure affecting link purpose, an ARIA-reference failure affecting link purpose, or a related redundancy/best-practice concern.

Recommendation: Label link-purpose-in-context explicitly as “Requires manual verification.” Do not present it as a confirmed WCAG 2.4.4 failure unless SiteLint identifies a specific link and a specific reason the link purpose cannot be determined from its text or programmatically determinable context. Broad manual prompts should be visually separated from concrete pattern-based review findings.

Recommendation: Treat img-alt-duplicate-text-link carefully. Duplicate image/link text may be noisy or redundant, but it is not always a clear WCAG 2.4.4 failure unless it actually interferes with understanding the link purpose.

Recommendation: Distinguish concrete automated findings, concrete pattern-based manual findings, and broad manual review prompts. links-same-content-different-url should be treated as a concrete pattern-based review candidate because it detects repeated accessible names with different destinations. The report should include each affected link, destination URL, selector or XPath, and a highlight option when possible. link-purpose-in-context should remain a broad manual reminder and should not be treated as evidence of automated WCAG 2.4.4 pass/fail coverage.
