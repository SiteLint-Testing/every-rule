WCAG 2.5.3: Label in Name

View main list

Brief summary: For user interface components with visible text labels, the accessible name should contain the visible label text. This helps speech-input users activate controls by speaking the label they can see.

View the W3C Understanding page for WCAG 2.5.3 Label in Name.

What SiteLint can test effectively

SiteLint's rule for this success criterion appears to compare a control's visible text label with the accessible name exposed through markup such as aria-label, aria-labelledby, native labels, or text content. This is useful automated coverage because the tool can often determine when visible label text is absent from the computed accessible name.

This is still partial coverage. The rule can evaluate common markup patterns, but full conformance may require judgment when the visible label is graphical, split across multiple elements, hidden by responsive layout, changed dynamically, or not straightforward to identify from the DOM.

Current SiteLint coverage expectation

SiteLint reports controls where the accessible name does not contain the visible label text, especially when the mismatch is caused by aria-label overriding visible text on buttons and links. SiteLint does not report controls where the visible label is contained in the accessible name or is the accessible name. A native input example that SiteLint does not currently catch is listed below in the “Not currently reported” section.

Recommendation for the current rule

Recommendation: Treat this as useful automated WCAG 2.5.3 coverage for common text-based controls, especially buttons and links with visible text overridden by mismatched aria-label values. The rule can catch many concrete mismatches between visible labels and accessible names, but it should not be presented as complete coverage for every possible visual-label scenario. The native input example should be considered a potential rule expansion area.

Additional scenarios and observed SiteLint behavior

Not currently reported by SiteLint

The following examples are relevant to WCAG 2.5.3. Some are not fully covered by SiteLint, while others were reported even though they were originally listed as manual/contextual scenarios. The distinction matters because clear visible-text/accesssible-name mismatches can often be tested automatically, while graphical, responsive, or ambiguous visual-label scenarios may still require review.

Not currently reported example 1: Input has a visible label but an overriding aria-label

This is a meaningful Label in Name risk because the visible label is “Phone number”, but the input's accessible name is overridden as “Mobile contact”. However, SiteLint did not report this scenario during testing, so it should not be listed as a current failing example for the rule.

Recommendation: Consider expanding the rule so it compares visible native label text against an overriding aria-label on the associated input. This is a clear text-label/accesssible-name mismatch, but it is not currently caught by SiteLint.

Not currently reported example 2: Visible label is graphical rather than text

The visible label may be an icon, symbol, or image of text. A DOM-based rule may not know what the visual label says unless the image or icon has reliable text equivalents.

Recommendation: SiteLint can verify that the control has an accessible name, but determining whether that name matches a visible icon or image may require image/icon interpretation or human judgment.

Additional observed failing examples SiteLint reports

Observed failing example 4: Initial-state visible label mismatch

SiteLint reports this example because the initial visible label is “Show details”, while the accessible name is “Expand details”. That is a clear initial-state Label in Name mismatch. The dynamic-state concern is separate: after activation, a control may change visible text or accessible name, and a static scan may not verify every later state.

This example also represents a control whose visible text could change from “Show details” to “Hide details” after activation.

Recommendation: Keep this as an observed failing example for the initial state. For complete dynamic-state coverage, SiteLint would still need interaction testing or DOM-change observation after controls are activated.

Not currently verified example 3: Responsive layout changes the visible label

A full label may be visible on desktop, while only an icon or abbreviated text appears on mobile. The accessible name may be correct in one viewport and mismatched in another.

This example represents a control whose visible text could be shortened or replaced by an icon at a narrow viewport.

Recommendation: A stronger rule would need to test multiple viewport sizes and compare the visible label and accessible name in each rendered state.

Additional observed failing examples continued

Observed failing example 5: Button text does not match accessible name

SiteLint reports this example because the button's own visible text is “Join now”, while its accessible name is “Subscribe”. Although there is nearby heading text, the direct button-text mismatch is clear enough for automated detection.

Newsletter subscription

The nearby heading provides context, but the visible label on the button itself is still “Join now”.

Recommendation: Keep this as an observed failing example. Ambiguous layout-based labels may still require manual review, but when the control itself has visible text that is absent from the accessible name, SiteLint can treat that as a concrete mismatch.

Recommended SiteLint treatment

Recommendation: Keep the current rule as automated WCAG 2.5.3 coverage for clear text-label/accesssible-name mismatches. It is stronger than a generic manual prompt because it can detect concrete mismatches in common controls. Expand the rule to catch visible native label text that is overridden by a mismatched aria-label on the associated input.

Future expansion could include dynamic-state testing, responsive viewport checks, OCR or icon-label analysis, and more nuanced handling for controls whose visible label is ambiguous or distributed across surrounding content.
