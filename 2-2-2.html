WCAG 2.2.2: Pause, Stop, Hide

View main list

Brief summary: Moving, blinking, scrolling, or auto-updating content that starts automatically, lasts more than five seconds, and is presented with other content generally needs a way for users to pause, stop, or hide it.

View the W3C Understanding page for WCAG 2.2.2 Pause, Stop, Hide.

This page separates two kinds of coverage: examples currently covered by SiteLint's animation.ts rule, followed by examples that are partially covered or outside the current rule's scope.

Examples currently covered by animation.ts

SiteLint's current animation rule appears to detect CSS animations applied to visible DOM elements when the animation is running and either lasts longer than five seconds or repeats infinitely. This is useful partial coverage of WCAG 2.2.2, but it does not fully determine whether a pause, stop, or hide mechanism exists or works.

Current SiteLint coverage expectation

SiteLint should report the three failing CSS-animation examples above. It should not report the four passing examples above. This confirms that the current rule is useful for detecting a narrow class of automatically moving content: CSS animations that are long-running or infinite.

Examples that are partially covered or outside the current animation.ts scope

The following examples demonstrate scenarios relevant to WCAG 2.2.2 where SiteLint may detect the animation pattern, but cannot always evaluate the full pass/fail condition. The current rule can identify long-running or infinite CSS animation. It does not fully verify whether the page provides a working pause, stop, or hide mechanism, or whether the motion is essential.

Scenario 1: Infinite scrolling text with no pause control

This is a likely WCAG 2.2.2 failure because the scrolling text starts automatically, repeats forever, and has no pause, stop, or hide control. SiteLint does detect this scenario through animation.ts because the scrolling text uses an infinite CSS animation with a duration longer than five seconds.

Breaking news: This announcement scrolls continuously without a pause, stop, or hide control.

Observed SiteLint result: SiteLint reports this scenario as an animation issue. This is appropriate for the current rule because the computed animation has a 12s duration and an infinite iteration count. The rule correctly catches the risky animation pattern, even though it is not proving the entire WCAG 2.2.2 failure condition by checking for the absence of a working pause, stop, or hide control.

Recommendation: Keep reporting this as an animation issue. This is useful partial WCAG 2.2.2 coverage. If the rule is expanded later, it should separately evaluate whether a pause, stop, or hide mechanism exists and works, rather than assuming the presence or absence of a nearby button is sufficient.

Scenario 2: Infinite scrolling text with a pause control

This may pass WCAG 2.2.2 if the pause control works and is available to keyboard and assistive technology users. The current animation.ts rule may still report the animation because it does not verify the relationship between the control and the moving content.

Service alert: This scrolling message includes a visible pause/resume control.

Recommendation: SiteLint should not automatically pass or fail this scenario from static markup alone. A stronger automated test would need to activate the pause button, confirm the animation stops, activate resume, and confirm the animation restarts. Without interaction testing, the tool can only identify this as a candidate for review or report the animation pattern.

Scenario 3: Auto-updating content

WCAG 2.2.2 also covers auto-updating content, not just CSS animation. This example updates a timestamp every second. The current animation.ts rule should not detect it because no CSS animation is involved.

Live dashboard value: Waiting for update...

This could fail WCAG 2.2.2 if the content updates automatically, is presented with other content, and users do not have a way to pause, stop, or hide the updates.

Recommendation: SiteLint could potentially expand coverage by detecting common auto-update patterns such as repeated DOM mutations, timers, or polling behavior. However, a reliable rule would need runtime observation, not a static DOM scan. It would also need to avoid false positives for harmless updates, such as clocks, timers that are essential, or content that updates only after user request.

Scenario 4: Blinking or moving content caused by JavaScript

A script can create blinking or moving content without using CSS animation-name. The current animation.ts rule should not detect this kind of behavior unless it results in computed CSS animation properties.

JS

Recommendation: SiteLint could expand coverage by observing style, class, or DOM changes over time and detecting repeated visual changes. This is possible in a browser-based auditor, but it is more complex than the current CSS animation check because the tool must observe time-based behavior and decide whether the visual change is significant enough to matter.

Scenario 5: Motion that lasts more than five seconds but is essential or user-triggered

Some motion may be allowed if it is essential, or if it does not start automatically. For example, an instructional animation that plays only after a user presses a button may not fail this SC in the same way as automatically moving content. The current animation.ts rule does not distinguish user-triggered animation from automatically running animation once the computed animation is active.

Demo

Recommendation: This is not a good candidate for a simple static rule. SiteLint would need to know whether the animation started automatically or only after a user request, and whether the animation is essential. The first question can be tested with runtime observation; the second often requires human judgment about the content's purpose.

Recommended SiteLint rule expansion

Recommendation: Keep animation.ts as a focused, non-manual rule for long-running and infinite CSS animations, but describe it as partial WCAG 2.2.2 coverage rather than full SC coverage.

Potential rule expansion should happen in stages. First, detect more CSS-based moving or blinking patterns and account for multiple animation durations and iteration counts. Second, add optional runtime checks for whether pause, stop, or hide controls actually change the animation state. Third, consider a separate rule for auto-updating regions, based on repeated DOM mutations or timer-driven updates. Full WCAG 2.2.2 conformance still requires human judgment in some cases because the tool may not know whether movement is essential, whether it is presented in parallel with other content, or whether a control is sufficient for the user's task.
