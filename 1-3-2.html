WCAG 1.3.2: Meaningful Sequence

View main list

Brief summary: When the sequence of content affects meaning, the correct reading sequence must be programmatically determinable. In practice, this often means the DOM/source order should match the meaningful reading order, even when CSS is used for visual layout.

View the W3C Understanding page for WCAG 1.3.2 Meaningful Sequence.

Caution: This success criterion is only partly testable through automation. A tool can identify some risky patterns, such as CSS reordering that makes visual order differ from DOM order, but determining whether that sequence changes meaning may still require human judgment.

Passing examples

Passing example 1: Step sequence appears visually and programmatically in the same order

This succeeds because the numbered steps appear in the same order visually and in the DOM. The sequence is meaningful and programmatically determinable.

  1. Step 1: Create the account.
  2. Step 2: Verify the email address.
  3. Step 3: Sign in to the dashboard.

Passing example 2: Cards retain a meaningful DOM order

This succeeds because the cards are presented in a meaningful sequence in the DOM: introduction first, details second, and summary last.

Failing examples

Failing example 1: CSS changes the visual order of meaningful steps

This fails because the visual order is changed with CSS order, so the steps appear visually as Step 2, Step 1, Step 3 while the DOM order remains Step 1, Step 2, Step 3. When sequence matters, the meaningful order should be programmatically determinable and should not rely only on visual CSS reordering.

SiteLint observation: SiteLint does not report this intentionally failing example. The page uses CSS order to make the visual sequence differ from the DOM sequence, which is a concrete, machine-detectable risk pattern for WCAG 1.3.2.

Recommendation: Add or improve a 1.3.2-oriented rule that detects CSS reordering on meaningful content, such as flex or grid items using non-default order values. The finding may still require human confirmation, but this pattern is specific enough to warrant at least a targeted warning or manual-verification item.

  1. Step 1: Create the account.
  2. Step 2: Verify the email address.
  3. Step 3: Sign in to the dashboard.

Failing example 2: CSS reorders content cards into a different visual sequence

This fails because CSS changes the visual order of related content. The details card is visually presented before the introduction card, but the DOM order still presents the introduction first. This creates a mismatch between visual reading order and programmatic reading order.

SiteLint observation: SiteLint does not report this intentionally failing example. The visual card order differs from the DOM order because CSS order is used, creating the same kind of meaningful-sequence risk in a layout pattern that is common on real sites.

Recommendation: SiteLint should consider flagging CSS-reordered flex or grid content as a 1.3.2 manual-verification issue when the visual order may differ from the programmatic reading order. This would help identify likely meaningful-sequence problems without overclaiming that every use of CSS reordering is automatically a failure.

Current 1.3.2 coverage takeaway

Recommendation: SiteLint does not currently appear to detect the two CSS-reordering scenarios on this page. These are not purely subjective examples: both use explicit CSS order values that make visual order differ from DOM order. Because 1.3.2 still depends on whether the changed sequence affects meaning, the safest reporting approach may be manual verification rather than a confirmed failure.

This page suggests a potential coverage gap for WCAG 1.3.2. SiteLint should consider a targeted rule for flex/grid reordering patterns where source order and visual order may diverge.
