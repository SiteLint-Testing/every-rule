WCAG 1.4.1: Use of Color

View main list

Brief summary: Color must not be the only visual means of conveying information, indicating an action, prompting a response, or distinguishing a visual element.

View the W3C Understanding page for WCAG 1.4.1 Use of Color.

What SiteLint can test effectively

SiteLint's links-not-visually-evident-without-color-vision rule appears to test one narrow, automatable part of WCAG 1.4.1: text links that are visually indistinguishable from surrounding text. The rule checks non-empty links and compares their computed style against the parent text. It can catch cases where the link has the same color, background, font style, and text decoration as the surrounding text.

This is useful partial coverage. It does not prove full WCAG 1.4.1 conformance because the success criterion applies to many other uses of color, including errors, statuses, chart legends, selected states, and instructions.

Current SiteLint coverage expectation

SiteLint should report the hidden-looking links because they are non-empty anchors inside ordinary paragraph text with inherited color and no text decoration. These examples use regular URL destinations rather than self-referencing fragment links so they are closer to ordinary link usage. SiteLint should not report the underlined link, the italic link, or the button-like link because those examples provide non-color visual cues or otherwise differ from the parent text in ways the current rule evaluates.

Recommendation for the current rule

Recommendation: Treat this rule as partial WCAG 1.4.1 coverage. The current rule is useful because it catches links that are effectively invisible as links. The rule name and reporting text should make clear that it is checking one link-specific pattern, not all possible uses of color.

Examples that are not fully covered by SiteLint's current 1.4.1 rule

The following examples are relevant to WCAG 1.4.1, but they are not reliably evaluated by SiteLint's current link-style comparison rule. Some patterns could be partially detected with specialized heuristics, but full conformance usually requires understanding the visual meaning, the instructions, or the relationship between color and content.

Scenario 1: Link distinguished only by color

This link is blue and not underlined. It is visually different from the surrounding text only by color. The current SiteLint rule may not report this because the link color differs from the parent text color, even though color is the only cue.

Recommendation: SiteLint could potentially expand link-specific 1.4.1 coverage by detecting non-underlined links whose only visual distinction from surrounding text is color. A stronger rule would also need to consider contrast between the link and surrounding text, hover/focus styling, and other non-color cues.

Scenario 2: Error information conveyed only by color

In this example, the instruction says red fields need attention, but the field itself has no icon, text, or programmatic error message. A human can infer that color is carrying meaning. A generic DOM scan cannot reliably know whether the color is meaningful or decorative without understanding the surrounding instructions and form state.

Fields shown in red need attention.

Recommendation: SiteLint could detect some risky patterns, such as red text near a form field without an error message, but it should not treat that alone as a confirmed 1.4.1 failure. Determining whether color is the only cue requires context about the instruction, the form state, and any visible or programmatic alternatives.

Scenario 3: Status conveyed only by red and green text

These statuses use color to distinguish success from failure. The words themselves are generic, so color is doing most of the semantic work. A tool can detect red and green text, but it cannot reliably determine whether the color is essential to understanding the status without evaluating the surrounding content.

Recommendation: SiteLint could flag common color-only status patterns as possible review items, especially red/green status indicators. However, confirming a WCAG 1.4.1 failure requires knowing whether another cue, label, icon, or text explanation also conveys the meaning.

Scenario 4: Chart legend distinguished only by color

Chart, map, and data-visualization examples are difficult to evaluate reliably with simple automation. The tool would need to understand that the colored swatches correspond to visual chart areas and that color is the only differentiator.

The selected tab below is indicated only by blue text. There is no underline, border, icon, text label, or other non-color cue. This can be a 1.4.1 problem, but a generic automated rule would need to infer that the visual state is meaningful and that color is the only visible cue.

Recommendation: SiteLint could detect some stateful components where the active state has only a color difference, but reliable detection requires comparing visual states and understanding component semantics. This is possible for some standard patterns, but it is not a simple page-wide static check.

Recommended SiteLint treatment

Recommendation: Keep the current link-style rule as a confirmed automated rule for one narrow 1.4.1 pattern: links that are visually indistinguishable from surrounding text. Do not present it as complete 1.4.1 coverage.

Future expansion could add specialized heuristics for non-underlined color-only links, color-only form errors, status indicators, selected states, and chart/data-visualization review candidates. Those should be framed carefully because most require context, visual comparison, interaction testing, or human judgment before they can be treated as confirmed WCAG failures.
