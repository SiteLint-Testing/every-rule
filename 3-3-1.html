WCAG 3.3.1: Error Identification

View main list

Brief summary: If an input error is automatically detected, the item in error must be identified and the error must be described to the user in text.

View the W3C Understanding page for WCAG 3.3.1 Error Identification.

Observed SiteLint behavior

SiteLint's error-identification rule reports this page as a manual verification item titled "Verify error identification" when the page contains visible form controls. The rule reports once at the page level rather than evaluating each error example.

This matches the current rule implementation, which selects visible input, select, textarea, and [role="textbox"] elements. If any matching element exists, the rule asks for manual verification. It does not determine whether aria-invalid is present, whether an error message exists, whether the error identifies the field, or whether the error text is programmatically associated with the input.

The page uses explicit autocomplete values on the inputs so the separate 1.3.5 autocomplete-attribute-usage rule does not obscure the 3.3.1 manual verification result.

Passing examples

Passing example 1: Error identified in text and associated with the field

The field is marked invalid and the text error is referenced with aria-describedby.

Enter an email address in the format name@example.com.

Passing example 2: Form-level error summary names the field

The error summary identifies the field in error and describes the problem in text.

There is a problem with the form.

  • ZIP code must contain five digits.

Failing examples

Failing example 1: Error shown only with color and border

The field appears to be in error, but no text identifies the error or describes what is wrong.

Failing example 2: Generic error text does not identify the item

The text says there is an error, but it does not identify which field is in error or describe the problem.

Please fix the errors below.

Manual review examples

Manual example 1: Error appears after submission

Some 3.3.1 issues can only be evaluated after a user submits a form or leaves a field.

Recommended SiteLint treatment

Recommendation: Treat the current error-identification rule as a broad manual-review trigger, not automated WCAG 3.3.1 evaluation. It is useful for reminding auditors to test forms, but it should not imply that the page's error identification has passed or failed.

To evaluate pages like this more effectively, SiteLint should inspect fields that are already in an error state, such as controls with aria-invalid="true", invalid native constraint states, or nearby visible error messaging. The rule should then check whether the field in error is identified in text and whether the error description is associated through mechanisms such as aria-describedby, an error summary link, or clear adjacent text.

SiteLint should continue to treat dynamic validation workflows as manual or interaction-based checks, because many 3.3.1 failures only appear after submission, blur, or other user actions.
