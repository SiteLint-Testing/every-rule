WCAG 4.1.2: Name, Role, Value

View main list

Brief summary: User interface components must expose their name and role, and states, properties, and values that can be set by the user must be programmatically determinable.

View the W3C Understanding page for WCAG 4.1.2 Name, Role, Value.

Observed SiteLint behavior

SiteLint correctly reports three 4.1.2 failures on this page: a focusable element inside aria-hidden="true", an empty button with no accessible name, and an iframe missing a title attribute.

SiteLint does not report the custom switch example that is focusable and visually presented as a switch but does not expose a switch role or checked state. That scenario is still relevant to WCAG 4.1.2 because custom controls need programmatically determinable roles, names, states, and values.

The page gives the empty-name button and other interactive examples sufficient visual dimensions so unrelated WCAG 2.5.8 target-size findings do not obscure the 4.1.2 results.

Notifications on

Observed result: SiteLint did not report this example. The element is keyboard focusable and visually presented as a switch-like control, but it is a generic span with no programmatic role and no checked state. Assistive technologies cannot reliably determine that it is intended to be a switch or whether it is on or off.

Recommendation: Add a 4.1.2 rule for focusable non-native controls that lack an appropriate role, name, or state. For switch-like controls, SiteLint should expect semantics such as role="switch" and aria-checked, or a native control that exposes an equivalent role and state. This can start as a high-confidence check for focusable generic elements styled or labeled like common widgets, with lower-confidence cases reported for manual review.

Manual example 1: Dynamic value changes

A static scan may not know whether a custom component updates its programmatic state after interaction.

Recommended SiteLint treatment

Recommendation: Preserve the current automated checks for focusable descendants inside aria-hidden="true", empty button names, and missing iframe titles. These are concrete, useful 4.1.2 failures.

Expand 4.1.2 coverage for custom controls. Focusable non-native controls that visually behave like switches, buttons, tabs, sliders, menus, or similar widgets should expose appropriate roles and states, such as role="switch" with aria-checked for a switch-like control.

Some dynamic value/state behavior still requires interaction testing. Static scans can identify missing initial roles and states, while interaction-based checks are needed to confirm that states update correctly after user actions.
