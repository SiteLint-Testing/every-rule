WCAG 4.1.1: Parsing

View main list

Brief summary: WCAG 4.1.1 Parsing required markup to be well-formed enough that user agents and assistive technologies could parse it reliably. In WCAG 2.0 and 2.1, this included requirements such as complete start/end tags, proper nesting, no duplicate attributes, and unique IDs.

View the W3C Understanding page for WCAG 4.1.1 Parsing.

Important WCAG version note

WCAG 4.1.1 Parsing was removed in WCAG 2.2. That means this success criterion should not be reported as an active WCAG 2.2 failure. However, the underlying technical problems can still matter. Duplicate IDs, malformed relationships, and invalid markup can still cause accessibility problems when they affect labels, ARIA references, fragment links, scripted widgets, or assistive technology exposure.

For SiteLint, this means the same detector may be useful in more than one reporting context. Under WCAG 2.0 or WCAG 2.1, duplicate IDs can map directly to 4.1.1. Under WCAG 2.2, generic duplicate IDs should not be reported as WCAG 4.1.1 failures because 4.1.1 has been removed. Reporting the generic duplicate-ID issue as a SiteLint / Best Practices finding is appropriate for WCAG 2.2, but when a user selects WCAG 2.0 or 2.1, the same issue should ideally be reportable under 4.1.1 Parsing. If a duplicate ID causes a specific accessibility failure under WCAG 2.2, that downstream problem should be reported under an active success criterion such as 1.3.1, 3.3.2, or 4.1.2.

Observed SiteLint classification

SiteLint currently reports the duplicate-ID findings on this page under the SiteLint / Best Practices classification. That is the correct treatment when the selected standard is WCAG 2.2, because 4.1.1 Parsing was removed from WCAG 2.2. However, when the selected standard is WCAG 2.0 or 2.1, this same duplicate-ID condition should ideally be classified under WCAG 4.1.1 Parsing rather than only as a Best Practices finding.

The rule is still useful because duplicate IDs can cause accessibility and scripting problems. The important product distinction is version-aware classification: under WCAG 2.0 or 2.1, the generic duplicate-ID finding can be a 4.1.1 issue; under WCAG 2.2, the generic duplicate-ID finding should depend on the Best Practices filter. It should not appear as a strict WCAG 2.2 issue unless another more specific rule identifies a current-SC failure caused by the duplicate ID.

Examples that are not useful to test as live HTML

Scenario 1: Duplicate attributes

WCAG 2.0 and 2.1 Parsing also included duplicate attributes. However, duplicate attributes are difficult to demonstrate as a live DOM test because browsers normalize invalid markup while parsing. The browser may discard one of the attributes before SiteLint inspects the DOM. 

<button type="button" type="submit">Duplicate type attributes</button>

Recommendation: If SiteLint wants to detect duplicate attributes, that should happen through source-code or HTML-string validation before the browser normalizes the markup. A DOM-only scan is not reliable for this specific parsing defect.

Scenario 2: Improperly nested elements or missing end tags

Improper nesting and missing end tags were also historically part of Parsing. They are not good candidates for this runtime DOM-based test page because the browser repairs the DOM before SiteLint evaluates it. 

<p>Paragraph starts
  <div>Invalidly nested div</div>
</p>

Recommendation: These issues are better handled by an HTML parser, validator, build-time tool, or source inspection. Runtime DOM inspection may see the repaired DOM rather than the original authoring error.

Recommended SiteLint treatment

Recommendation: Do not maintain separate overlapping duplicate-ID rules for WCAG 2.1 and WCAG 2.2. Keep one underlying duplicate-ID detector, then make the reporting classification version-aware.

In WCAG 2.0 or 2.1 mode, report duplicate IDs under Accessibility → WCAG 4.1.1 Parsing. In WCAG 2.2 mode, do not report generic duplicate IDs as WCAG 4.1.1. The current SiteLint approach of reporting them as SiteLint / Best Practices findings is appropriate for WCAG 2.2, but not ideal when WCAG 2.0 or 2.1 is the selected target. If Best Practices are disabled in a WCAG 2.2-only review, generic duplicate-ID findings should be hidden unless a duplicate ID breaks a specific accessibility relationship and can be reported under an active WCAG 2.2 success criterion.

For the product filters, the strongest model is: Audit group: Accessibility, Rule type: Best Practice or Accessibility Risk for WCAG 2.2, and WCAG mapping: 4.1.1 for WCAG 2.0 and 2.1.
