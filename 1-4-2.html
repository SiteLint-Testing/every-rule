WCAG 1.4.2: Audio Control

View main list

Brief summary: If audio plays automatically for more than three seconds, users need a way to pause or stop it, or a way to control its volume independently from the system volume.

View the W3C Understanding page for WCAG 1.4.2 Audio Control.

This page separates examples SiteLint can evaluate effectively from examples that require additional runtime, media, or human judgment. SiteLint's current autoplay rule detects explicit autoplay patterns such as an autoplay attribute or iframe URLs containing autoplay=1. That is useful partial coverage of WCAG 1.4.2, but it is not full conformance testing.

Important testing note: These examples use placeholder media paths and about:blank iframe sources so the page can test markup patterns without requiring real media assets. That is enough to test whether SiteLint detects autoplay attributes and parameters. It is not enough to test actual playback behavior, media duration, audibility, or whether the reported media elements can be reliably highlighted in the page UI.

This should be reported by SiteLint because the element has an explicit autoplay attribute. This is a high-risk WCAG 1.4.2 pattern because audio may start without user request.

Failing example 2: Video element has autoplay

This should be reported by SiteLint because the element has an explicit autoplay attribute. This is relevant to WCAG 1.4.2 when the video includes audio that starts automatically.

This should be reported by SiteLint because the iframe src contains autoplay=1, which commonly indicates embedded media configured to play automatically.

This should be reported if SiteLint checks the current selector iframe[allow*="autoplay=1"]. This is a narrow iframe-permission pattern.

Current SiteLint coverage expectation

SiteLint reports the examples with autoplay and autoplay=1. It does not report the examples without autoplay declarations. This confirms useful partial coverage: the rule can identify explicit autoplay declarations directly from the DOM.

In the reviewed result, SiteLint reported six autoplay issues. That is expected for this page: the four direct failing examples plus the muted autoplay video and autoplaying audio with controls in the later partial-coverage section. The current rule flags the autoplay pattern itself; it does not make the full WCAG judgment about audibility, duration, or whether controls satisfy the requirement.

Observed secondary SiteLint findings

This page can also trigger SiteLint findings for media alternatives, such as captions, descriptions, or transcripts. Those findings are related to separate WCAG media requirements, not to WCAG 1.4.2 Audio Control.

This is not necessarily rule overlap or a mapping error. A single audio, video, or media iframe element can legitimately trigger more than one accessibility rule. For example, autoplay may be relevant to WCAG 1.4.2, while missing captions or descriptions may be relevant to WCAG 1.2.x. For this test page, the secondary media-alternative findings are acceptable and should be expected, but they should be interpreted separately from the autoplay findings.

Recommendation: The rules do not need to be merged or removed simply because the same element triggers multiple issues. However, SiteLint should make the WCAG mapping, issue title, and remediation guidance clear enough that users understand these are separate problems on the same media element, not duplicate reports for the same requirement.

Observed element-highlighting behavior

In the reviewed result, the reported audio, video, and iframe elements did not reliably highlight on the page. SiteLint should ideally be able to highlight the affected element whenever the element has a rendered box. If highlighting is not possible or is unreliable for native media controls, empty iframes, or placeholder media, SiteLint should provide a fallback message that explains the element cannot be visually highlighted and directs the user to the selector, DOM path, or source snippet.

The test page may also be improved later by referencing actual short media files and real embedded media examples. Real media would help evaluate playback behavior, duration, controls, and highlighting against rendered media. Placeholder paths are still appropriate for testing static autoplay detection, but they are not sufficient for testing the complete user experience of locating and evaluating media playback issues.

Recommendation: Keep this page focused on markup-level autoplay detection for now, but improve the test page later with actual media files and rendered iframe content so SiteLint highlighting can be demonstrated and evaluated. If the goal expands to testing playback behavior, create a separate version that uses real media assets with known durations, muted and unmuted variants, and working controls.

Examples that are detected but only partially evaluated, or outside the current rule's scope

WCAG 1.4.2 is not fully determined by the presence of an autoplay attribute. A complete evaluation requires knowing whether audible audio actually starts automatically, whether it continues for more than three seconds, and whether users have a working way to pause, stop, mute, or independently control volume. Some examples below are detected by SiteLint because they contain autoplay markup, but they still require additional context before they can be treated as confirmed WCAG 1.4.2 failures.

Scenario 1: Muted autoplaying video

This video has an autoplay attribute, but it is also muted. WCAG 1.4.2 is specifically about audio. If no audio is played, this may not be a true 1.4.2 failure, though it may still be relevant to other motion or distraction concerns.

Observed SiteLint result: SiteLint reports this because it contains the autoplay attribute. That is expected under the current rule, but the result needs nuance because the media is muted.

Recommendation: SiteLint should consider checking whether autoplaying video is muted. A muted autoplay video should not automatically be treated as a confirmed WCAG 1.4.2 failure, although it may still be useful to report as a caution or evaluate under a separate motion-related rule.

Scenario 2: Autoplaying audio with controls

This audio element has both autoplay and controls. The current rule should report the autoplay attribute. A full WCAG 1.4.2 judgment requires determining whether the controls are available quickly enough and whether the audio lasts more than three seconds.

Observed SiteLint result: SiteLint reports this because it contains the autoplay attribute. That is expected under the current rule. The presence of native controls does not prove compliance unless the controls are available and sufficient for the user before or during playback.

Recommendation: Keep reporting this as an autoplay issue, but do not claim full WCAG failure from markup alone. SiteLint could improve the message by saying the rule detects autoplay and that duration and control behavior may require verification.

Scenario 3: Audio is shorter than three seconds

WCAG 1.4.2 applies when audio plays automatically for more than three seconds. Static HTML does not reveal the media duration unless the browser loads metadata or the tool inspects the media file.

Example: <audio autoplay src="media/two-second-sound.mp3"></audio>

Recommendation: SiteLint could expand coverage by loading media metadata and checking duration when feasible. This is possible in some browser contexts, but it may fail for cross-origin media, blocked resources, streaming content, or media that does not expose duration reliably.

Scenario 4: Iframe allows autoplay but does not actually autoplay

A common iframe permission pattern is allow="autoplay; fullscreen". This grants autoplay permission, but it does not necessarily mean the embedded media starts automatically. The current selector appears to look for allow*="autoplay=1", which is narrower than this common pattern.

Recommendation: Do not automatically fail all iframes with allow="autoplay". SiteLint could flag them as a caution only when the iframe also appears to request autoplay through its URL or known provider parameters. The mere permission is not enough to prove audio starts automatically.

Scenario 5: Autoplay starts only after user interaction

Modern browser policies often block audible autoplay until the user interacts with the page. A page may include autoplay markup, but actual playback behavior may depend on the browser, media state, user settings, and prior interaction.

Example: a media player script calls play() only after the user presses a button or after browser permission is granted.

Recommendation: This cannot be fully determined by static markup. A stronger test would need runtime playback observation: load the page, avoid user interaction, observe whether audible media starts, and determine whether playback continues for more than three seconds.

Scenario 6: Volume control is custom or independent from system volume

WCAG 1.4.2 allows a mechanism to control audio volume independently from the overall system volume. A page may provide custom media controls that satisfy this requirement, but a static scan cannot reliably determine whether those controls work.

Example: a custom player has buttons labeled Mute, Pause, and Volume, but the tool would need interaction testing to confirm those controls affect the audio.

Recommendation: SiteLint could identify candidate controls by accessible names such as “mute,” “pause,” or “volume,” but it should not treat their presence as proof of compliance. Confirming this requires activating controls and observing playback or volume changes.

Recommended SiteLint rule treatment

Recommendation: Treat the autoplay rule as partial WCAG 1.4.2 coverage. It is valuable because explicit autoplay declarations are machine-detectable and high risk. The rule should avoid claiming full conformance coverage because it does not currently confirm audibility, duration, working controls, iframe playback behavior, or independent volume control.

Useful expansions would include detecting muted autoplay separately, improving iframe parameter coverage, checking media duration when metadata is available, improving visual highlighting or fallback location messaging for native media and iframe elements, and optionally adding runtime playback checks. Some aspects will still require manual or interactive verification because the tool must determine whether audio actually plays, whether user controls work, and whether custom controls satisfy the SC.
