WCAG 2.4.6: Headings and Labels

View main list

Brief summary: Headings and labels should describe the topic or purpose of the content, section, field, or control they identify.

View the W3C Understanding page for WCAG 2.4.6 Headings and Labels.

What this page is intended to test

This page provides a first pass at WCAG 2.4.6 scenarios involving headings, form labels, button labels, and link labels. Some scenarios are directly testable when text is empty or obviously generic. Other scenarios require judgment because a heading or label may be technically present but still not describe the topic or purpose clearly enough.

Initial testing should identify which SiteLint rules are mapped to WCAG 2.4.6, which examples are caught automatically, which examples are reported under related criteria, and which examples require manual review.

Failing or risky examples

Failing example 1: Generic heading does not describe the section

A heading such as “Information” may be too vague to describe the topic or purpose of the section.

Information

Hotel booking instructions, group-rate deadlines, and venue transportation details are listed here.

Recommendation: SiteLint rules should be improved to catch generic headings when they do not describe the section topic clearly enough. This may need to be a review candidate rather than a confirmed failure because context matters.

Failing example 2: Empty heading

An empty heading creates structure without describing any topic.

This content appears after an empty heading.

Failing example 3: Label is present but not descriptive

The label exists, but “Info” does not clearly describe the expected input.

Recommendation: SiteLint rules should be improved to catch labels that are technically present but too vague to describe the purpose of the field.

Secondary example 4: Placeholder is used instead of a persistent label

Placeholder-only labeling can disappear when the user types and may not provide a persistent label for the field. This is relevant to label quality, but the live input triggers an unrelated WCAG 3.3.2 “Form controls need labels” finding, so the example is shown as a code sample on this page.

<input id="placeholder-only-phone" name="placeholder-only-phone" type="tel" placeholder="Phone number" autocomplete="tel">

Recommendation: SiteLint should identify inputs that rely on placeholder text instead of a persistent label where that pattern creates a label-purpose risk, while keeping the finding taxonomy separate from missing-label coverage under WCAG 3.3.2.

Failing example 5: Repeated generic buttons are not distinguishable

Repeated buttons with the same generic label can be unclear when users encounter them out of context or when the surrounding card context is not programmatically associated.

Attendee registration

Register to attend the conference.

Exhibitor registration

Register to reserve an exhibitor booth.

Recommendation: SiteLint rules should be improved to catch repeated generic control labels when the surrounding context is not sufficient or not programmatically determinable.

Secondary example 6: Link label is generic and not meaningful

Generic link text can fail heading-and-label expectations when the label does not describe the destination or action clearly enough. This overlaps with link-purpose rules, and the live link triggers an unrelated WCAG 2.4.9 finding, so the example is shown as a code sample on this page.

<a href="https://example.com/session-details">Click here</a>

Recommendation: SiteLint should keep generic link-label findings clearly separated between WCAG 2.4.6, 2.4.4, and 2.4.9 so this page can test heading-and-label clarity without creating unrelated link-purpose noise.

Observed SiteLint result

SiteLint currently reports only failing example 2 as a WCAG 2.4.6 issue: “Heading element exists but has no content.” It does not report failing example 1, failing example 3, or failing example 5 as WCAG 2.4.6 findings, even though those examples represent headings or labels that are present but not descriptive enough.

The placeholder-only input originally triggered an unrelated WCAG 3.3.2 “Form controls need labels” finding, and the generic link originally triggered an unrelated WCAG 2.4.9 finding. Those examples have been converted to code samples so the live page remains focused on WCAG 2.4.6 findings.

Recommended SiteLint treatment

Recommendation: Treat empty headings as confirmed automated WCAG 2.4.6 findings. SiteLint already catches the empty heading scenario on this page.

Recommendation: Improve SiteLint rules to catch or review headings and labels that are present but not descriptive enough. Failing example 1 uses a generic heading, failing example 3 uses a vague form label, and failing example 5 uses repeated generic button labels. These should be treated as review candidates because context affects whether the heading or label adequately describes the topic or purpose.

Recommendation: Keep the distinction clear between WCAG 2.4.6 label/heading clarity and related link-purpose findings under WCAG 2.4.4 or 2.4.9. When the same issue can map to multiple criteria, SiteLint should explain the classification and avoid duplicate or contradictory reports.
