WCAG 1.4.4: Resize Text

View main list

Brief summary: Except for captions and images of text, text should be resizable up to 200 percent without loss of content or functionality.

View the W3C Understanding page for WCAG 1.4.4 Resize Text.

What SiteLint can test effectively

SiteLint's zoom-disabled rule appears to test one narrow, automatable part of WCAG 1.4.4: whether the page restricts browser zoom through the viewport meta tag. This is useful partial coverage because disabling zoom can prevent users from enlarging text enough to read it.

This test page intentionally includes user-scalable=no in the document head. SiteLint should report that setting because it restricts user zoom. However, the rule does not fully determine whether all text can be resized to 200 percent without clipping, overlap, or loss of functionality.

Current SiteLint coverage expectation

SiteLint should report the active viewport meta tag in the document head because it includes user-scalable=no. SiteLint should not report the code samples shown in the body because those are visible examples, not active viewport metadata.

Observed highlighting behavior

The reported viewport meta element cannot be visually highlighted in the rendered page because it lives in the document head and does not render a visible box in the page body. SiteLint should provide a fallback location, such as a DOM path, source snippet, or DevTools location, rather than depending on visual highlighting.

Recommendation for the current rule

Recommendation: Treat zoom-disabled as useful partial WCAG 1.4.4 coverage. It detects viewport settings that disable or restrict zoom, but it does not verify that text can actually be resized to 200 percent without clipping, overlap, or loss of functionality.

The rule should also detect other restrictive viewport patterns, such as user-scalable=0, very low maximum-scale values, or minimum-scale / maximum-scale combinations that do not allow sufficient zoom. The rule should compare viewport values case-insensitively and report non-rendered head elements with an appropriate UI fallback.

Examples that are not fully covered by SiteLint's current 1.4.4 rule

The following examples are relevant to WCAG 1.4.4, but they are not evaluated by a simple viewport-zoom check. They require actually increasing text size or browser zoom and verifying that content and functionality remain usable.

Scenario 1: Text clips inside a fixed-height container

This box uses a fixed height and hidden overflow. At larger text sizes or browser zoom levels, some text may be clipped. SiteLint's current viewport rule should not detect this because the issue is layout behavior after resizing, not a viewport meta setting.

This text is inside a fixed-height container with hidden overflow. At 200 percent text size, content may be cut off.

Recommendation: SiteLint could potentially create a separate runtime layout test that increases zoom or text size and then checks for clipped or hidden content. That requires browser rendering and comparison before and after resizing.

Scenario 2: Text overlaps when resized

Absolute positioning can cause text to overlap when zoomed or enlarged. SiteLint's current viewport rule should not detect this because the viewport allows or blocks zoom; it does not inspect layout reflow at 200 percent.

First line of positioned text.

Second line may overlap when text is enlarged.

Recommendation: Detecting overlap after resizing requires a rendered-page comparison at different zoom or text-size settings. Some cases can be automated with layout measurement, but a human may still need to verify whether the overlap causes loss of content or functionality.

Scenario 3: Very small fixed-size text

This text is intentionally small. Small text may be a usability concern, but WCAG 1.4.4 is specifically about whether text can be resized up to 200 percent without loss of content or functionality. The current viewport rule does not evaluate this text directly.

This is very small fixed-size text. It may be difficult to read, but the relevant 1.4.4 question is whether users can resize it successfully.

Recommendation: Very small text may be better handled by a separate readability or minimum-font-size best-practice rule. It should not be treated as a complete 1.4.4 failure unless resizing is restricted or causes loss of content/functionality.

Scenario 4: Controls become unusable after zoom

Text resizing can affect controls, navigation, dialogs, and forms. A page may allow zoom but still fail if controls overlap, disappear, or become unusable at 200 percent. SiteLint's current viewport rule does not test that interaction state.

Recommendation: This requires testing the page at increased zoom/text size and verifying that controls remain visible and operable. Automation can help with screenshots, layout bounds, and interaction checks, but human review may still be needed for complex responsive behavior.

Recommended SiteLint treatment

Recommendation: Keep zoom-disabled as a confirmed automated rule for viewport zoom restrictions. It is appropriate partial WCAG 1.4.4 coverage, but should not be presented as complete Resize Text coverage.

Future expansion could include a separate rendered-page test that evaluates layout at 200 percent zoom or increased text size, checks for clipping and overlap, and verifies that controls remain operable. That would be a different kind of runtime/responsive-layout test from the current viewport metadata check.
