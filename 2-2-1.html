WCAG 2.2.1: Timing Adjustable

View main list

Brief summary: When content imposes a time limit, users generally need a way to turn off, adjust, or extend the limit unless one of the WCAG exceptions applies.

View the W3C Understanding page for WCAG 2.2.1 Timing Adjustable.

What SiteLint can test effectively

SiteLint's no-meta-http-equiv-refresh rule appears to test one narrow, automatable part of WCAG 2.2.1: pages that use <meta http-equiv="refresh"> to refresh or redirect after a time interval. This is useful partial coverage because a meta refresh creates a time-triggered page change that users usually cannot pause, extend, or adjust from the page UI.

This page includes two intentionally long-delay meta refresh tags in the document head. The delay is set to 999999 seconds so the page does not redirect during normal testing, while still allowing SiteLint to detect the markup pattern.

Examples where SiteLint should be effective

Passing examples that should not be reported

Passing example 1: Ordinary metadata without refresh

The page includes normal metadata such as charset, viewport, and title. Those should not be reported by the meta-refresh rule because they do not impose a time limit. 

<meta charset="utf-8">
<meta name="viewport" content="width=device-width, initial-scale=1">
<title>Example page</title>

Passing example 2: Refresh-like words in visible text

This visible text mentions refresh behavior, but it is not a meta refresh tag. SiteLint should not report visible prose simply because it contains the word “refresh.”

Select the refresh button only when you want to manually reload the current data.

Failing examples that should be reported

Failing example 1: Meta refresh without redirect

This page's head includes the following meta refresh tag. SiteLint should report it because it automatically refreshes the current page after a time interval. 

<meta http-equiv="refresh" content="999999">

Failing example 2: Meta refresh redirect

This page's head also includes the following meta refresh redirect. SiteLint should report it because it automatically changes the user's context after a time interval. 

<meta http-equiv="Refresh" content="999999; url=https://example.com/redirect-target">

Current SiteLint coverage expectation

SiteLint reports the two meta refresh tags in the document head. The first uses http-equiv="refresh" and the second uses http-equiv="Refresh". SiteLint does not report ordinary metadata or visible text that merely references refreshing.

Observed highlighting behavior

SiteLint correctly reports the two meta refresh issues, but these elements cannot be visually highlighted in the rendered page because meta elements live in the document head and do not render visible boxes in the page body.

This is expected behavior for this type of finding. The report should rely on the DOM path, source snippet, or DevTools location rather than a visual page highlight. For non-rendered elements such as meta, title, script, or other head content, SiteLint should provide a clear fallback message indicating that the issue is not visually highlightable on the page.

Recommendation for the current rule

Recommendation: Treat this rule as useful partial WCAG 2.2.1 coverage. It detects an objectively risky time-based pattern, but it should not be presented as complete Timing Adjustable coverage because many time limits are implemented without meta refresh.

The rule should ideally normalize the http-equiv value case-insensitively so values such as REFRESH or ReFrEsH are detected consistently.

For reported head elements, the SiteLint UI should not depend on visual highlighting. A better user experience would explain that the element is non-rendered and provide the DOM path and source location as the primary way to locate the issue.

Examples that are not fully covered by SiteLint's current 2.2.1 rule

The following examples are relevant to WCAG 2.2.1, but they are not evaluated by a simple meta-refresh check. Some can be partially detected with runtime observation or JavaScript analysis, but full evaluation often requires understanding the purpose of the time limit and whether the user can turn it off, adjust it, or extend it.

Scenario 1: JavaScript-based timeout

A page can impose a time limit using JavaScript instead of meta http-equiv="refresh". SiteLint's current meta-refresh rule should not detect this because there is no meta refresh tag. 

setTimeout(function () {
  window.location.href = '/session-expired';
}, 300000);

Recommendation: SiteLint could potentially detect some JavaScript timeout patterns, but static JavaScript analysis can be brittle. A stronger approach would require runtime observation or instrumentation to detect time-triggered redirects, form resets, session expirations, or content changes.

Scenario 2: Session timeout with an extend option

A session timeout may pass WCAG 2.2.1 if users are warned and given enough time to extend the session. A simple scan cannot determine whether the warning appears at the right time, whether the extend control works, or whether data is preserved.

Session notice: Your session will expire soon.

Recommendation: SiteLint could identify candidate timeout dialogs and controls, but confirming WCAG conformance requires interaction testing: wait for the warning, activate the extension, and verify that the time limit is actually extended and user data is preserved as required.

Scenario 3: Timed quiz or task

Some pages include visible timers for quizzes, applications, or transactions. Whether the timer is allowed depends on the purpose of the time limit and whether one of the WCAG exceptions applies.

Practice quiz timer: 04:59 remaining

Recommendation: SiteLint may be able to identify timer-like text or controls as review candidates, but determining whether the time limit is essential, legally required, longer than 20 hours, adjustable, or extendable requires context and human judgment.

Scenario 4: Auto-updating content without a page refresh

Content can update on a timer without refreshing or redirecting the page. Depending on the behavior, this may overlap with WCAG 2.2.1 Timing Adjustable, WCAG 2.2.2 Pause, Stop, Hide, or both.

Live data: This example represents content that refreshes in place every 30 seconds.

Recommendation: A separate runtime rule could observe repeated DOM changes or polling behavior. However, the tool would still need to determine whether the updates impose a time limit, whether users can pause or adjust them, and which WCAG success criterion is the best fit.

Recommended SiteLint treatment

Recommendation: Keep no-meta-http-equiv-refresh as a confirmed automated rule for meta refresh and meta refresh redirects. It is an appropriate WCAG 2.2.1 rule for that narrow pattern.

Future expansion could add separate checks for JavaScript timeouts, session-expiration dialogs, timer widgets, and auto-updating content. Those should be framed carefully because complete 2.2.1 evaluation usually requires runtime observation, interaction testing, and judgment about the purpose of the time limit and the availability of turn-off, adjust, or extend mechanisms.
