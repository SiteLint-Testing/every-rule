WCAG 1.3.1: Info and Relationships

View main list

Brief summary: Information, structure, and relationships that are conveyed visually should also be programmatically determinable or available in text. Common examples include headings, lists, form labels, grouped controls, and table headers.

View the W3C Understanding page for WCAG 1.3.1 Info and Relationships.

These examples focus on structural relationships that an automated accessibility tool may reasonably evaluate from HTML. Broader 1.3.1 issues that depend mainly on visual presentation, author intent, or human interpretation are intentionally omitted from this page.

Failing examples

Failing example 1: Visible form label is not programmatically associated with the input

This fails because the text label appears visually next to the input, but it is not a real <label> and is not programmatically associated with the control.

SiteLint observation: SiteLint correctly reports this unlabeled input, but the finding is mapped to WCAG 3.3.2 rather than 1.3.1. The failure is relevant to both labels/instructions and programmatic relationships, so this is useful detection but incomplete evidence of direct 1.3.1 coverage.

Recommendation: Consider whether this rule should also be mapped to 1.3.1, or whether SiteLint should have a separate 1.3.1-specific form-label relationship rule. The user-facing message should make clear that the visible label is not programmatically associated with the control.

Failing example 2: Radio buttons are visually grouped but have no programmatic group label

This fails because the radio buttons are visually presented as a group, but there is no <fieldset> and <legend> or equivalent programmatic group label.

SiteLint observation: SiteLint correctly reports this grouped-control failure under WCAG 1.3.1, but the rule message says “Checkboxes should be grouped together” even though this example uses radio buttons. The underlying issue is correct, but the message is too narrow or inaccurate for radio groups.

Recommendation: Update the rule title/message to cover grouped controls generically, such as “Related checkboxes or radio buttons should be grouped together.” The rule should avoid saying “checkboxes” when the failing control group is actually a radio group.

SiteLint observation: SiteLint also reports a separate WCAG 2.5.8 target-size issue on the radio buttons because the native controls render smaller than 24 by 24 CSS pixels. That is outside the purpose of this 1.3.1 test page.

Recommendation: For a cleaner 1.3.1 test page, consider styling the radio controls or their clickable labels so the 2.5.8 finding does not distract from the grouping relationship being tested.

Failing example 3: Data table visually has headers but uses only data cells

This fails because the first row is visually acting as a header row, but it uses <td> cells instead of header cells. The column-header relationships are not programmatically determinable.

SiteLint observation: SiteLint correctly reports this table-header relationship failure under WCAG 1.3.1. This is strong evidence of useful automated coverage for at least one common 1.3.1 table-structure pattern.

Recommendation: Keep this as a strong automated 1.3.1 test scenario. It is a clear, machine-detectable relationship problem: a data table is present, but no header cells are defined.

Current 1.3.1 coverage takeaway

Recommendation: SiteLint shows useful 1.3.1-related coverage for some structural relationships, especially grouped controls and table headers. The first pass also shows two issues to address: the unlabeled input is reported under 3.3.2 rather than 1.3.1, and the grouped-control rule message should be generalized beyond “checkboxes.”

1.3.1 is broad, so this page intentionally focuses on machine-detectable structural relationships rather than scenarios that mostly depend on visual presentation or author intent.
