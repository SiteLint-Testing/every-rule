WCAG 2.4.1: Bypass Blocks

View main list

Brief summary: Users need a way to bypass repeated blocks of content, such as repeated navigation, so they can move directly to the main content or other important regions.

View the W3C Understanding page for WCAG 2.4.1 Bypass Blocks.

Passing examples

Passing example 1: Page has a skip link to main content

This page includes a skip link at the beginning of the body that targets #main-content. The link becomes visible on focus and moves the user past repeated content.

<a class="skip-link" href="#main-content">Skip to main content</a>

Passing example 2: Main landmark provides a programmatic bypass target

A properly exposed main landmark can help assistive-technology users move directly to the main content region.

<main id="main-content">...</main>

Passing example 3: Headings help users bypass sections

A clear heading structure can provide another mechanism for bypassing and navigating blocks of content, especially for screen reader users who navigate by headings.

Example content section

This section has a heading that identifies the content that follows.

Failing example 2: Skip link points to a missing target

A skip link is only useful if its target exists and moves focus or reading position to the intended content. This example is described rather than implemented because an actual broken fragment link triggers SiteLint's separate 2.4.4 same-page-link rule instead of the 2.4.1 bypass-blocks rule.

<a href="#missing-main-target">Skip to main content</a>

The target #missing-main-target would not exist in this example.

Failing example 3: Skip link is not visible on focus

Some pages include a skip link that remains visually hidden even when focused. That can make the mechanism difficult or impossible for sighted keyboard users to use.

This example is described rather than implemented because an actually invisible focused skip link would create an intentional keyboard-accessibility problem on this test page.

Observed SiteLint result

SiteLint reports a WCAG 2.4.1 bypass-blocks manual-verification item when it detects visible navigation-like structures such as nav, header, [aria-label="Skip links"], .main-navigation, or .skip-links. This page now includes a real nav element so the current rule should trigger.

The current rule does not actually determine whether the page passes or fails WCAG 2.4.1. It selects navigation-like elements and reports a manual-verification prompt. It does not check whether a valid skip link exists, whether the skip link target exists, whether the skip link becomes visible on focus, whether the page has a main landmark, or whether the available bypass mechanism is sufficient.

The earlier broken skip-link example was converted to a code sample because an actual broken fragment link triggers SiteLint's separate same-page-link rule under WCAG 2.4.4. That separate rule can be relevant to bypass behavior, but it is not the same as automated WCAG 2.4.1 coverage.

Recommended SiteLint treatment

Recommendation: Keep the current bypass-blocks behavior as a manual-verification prompt, not a confirmed automated failure. The rule is useful as a reminder when repeated navigation or header-like structures are present, but the report should make clear that human verification is required.

Recommendation: Add separate automated or semi-automated checks for concrete bypass-blocks patterns: missing main landmark, missing skip link on pages with repeated navigation, skip link target missing, skip link not visible on focus, multiple repeated navigation blocks without a bypass mechanism, or a bypass link that does not move the user to the intended content.

Recommendation: Keep broken skip-link-target detection under the same-page-link rule unless SiteLint intentionally cross-references it as a possible bypass-blocks problem. A confirmed 2.4.1 finding should identify the specific missing or broken bypass mechanism rather than reporting a generic page-level failure.
