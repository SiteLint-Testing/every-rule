WCAG 2.4.9: Link Purpose (Link Only)

View main list

Brief summary: The purpose of each link should be identifiable from the link text alone, except where the link purpose would be ambiguous to users in general.

View the W3C Understanding page for WCAG 2.4.9 Link Purpose (Link Only).

What this page is intended to test

WCAG 2.4.9 is stricter than WCAG 2.4.4. Under 2.4.4, link purpose may be determined from link text together with programmatically determinable context. Under 2.4.9, the link purpose generally needs to be clear from the link text alone.

This page tests generic link text, repeated link text, descriptive link text, and cases where surrounding context helps visually but the link text alone remains insufficient. Initial testing should identify whether SiteLint treats these as automated findings, manual-verification findings, or related 2.4.4 / 2.4.6 link-label issues.

Passing examples

Passing example 1: Link text describes the destination

This link should pass because the link text identifies the destination without relying on surrounding context.

View the conference schedule

Passing example 2: Download link identifies the file purpose

The link text explains what the user will download.

Download the accessibility checklist PDF

Passing example 3: Repeated links remain distinguishable

These links use similar action wording but identify different destinations from the link text alone.

Register as an attendee Register as an exhibitor Register as a sponsor

Failing or risky examples

Failing example 1: Generic “click here” link

The link text does not describe the destination or action when read by itself.

For the full conference agenda, click here.

Failing example 2: Generic “learn more” link

The surrounding sentence gives visual context, but the link text alone does not identify what the user will learn more about.

Accessibility training is available for presenters. Learn more.

Failing example 3: Repeated “Read more” links point to different destinations

Each article provides visual context, but the links themselves have identical text. A links list would expose several identical “Read more” links without enough information to distinguish them.

Accessibility training announced

New training sessions are available for content authors.

Read more

Conference registration opens

Registration is now open for this year's conference.

Read more

Failing example 4: Same link text points to different destination types

The links all say “Details,” but each points to a different type of information. The link text alone does not identify the purpose.

Details Details Details

Failing example 5: URL text is not meaningful enough

A bare URL may not clearly describe the link purpose, especially when the path is ambiguous or truncated visually.

https://example.com/x/93821

Recommendation: SiteLint should be improved to catch or review bare URL link text when the URL itself does not clearly identify the destination or purpose. This should likely be treated as a review candidate rather than an automatic failure because some visible URLs may be meaningful enough.

Observed SiteLint result

SiteLint caught failing examples 1, 2, 3, and 4 as WCAG 2.4.9 link-purpose issues. It did not report failing example 5, where the link text is a bare URL with an ambiguous path: https://example.com/x/93821.

The missed example may be intentional if SiteLint treats visible URL text as sufficiently descriptive by default, but this example demonstrates a link-purpose risk because the URL path does not identify the destination or purpose in human-readable terms.

Recommended SiteLint treatment

Recommendation: Treat obviously generic link text such as click here, learn more, read more, and details as strong candidates for WCAG 2.4.9 review because the link purpose is not clear from the link text alone. Also consider bare URL text with ambiguous paths as a review candidate when the URL does not communicate a human-readable purpose.

Recommendation: Distinguish WCAG 2.4.9 from WCAG 2.4.4. A link may satisfy 2.4.4 when its purpose is clear from programmatically determinable context, while still failing or requiring review under 2.4.9 because the link text alone is not specific enough.

Recommendation: When SiteLint reports repeated generic links, the report should list each affected link, destination URL, selector or XPath, and provide a highlight option when possible.
